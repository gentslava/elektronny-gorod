# Карточка звонка с домофона — Production UX

**Интеграция:** Электронный Город (Home Assistant · Lovelace)
**Компонент:** `<eg-intercom-call-card>` (Lit + TypeScript)
**Макеты:** `pencil/design.pen` — 6 секций (мобайл тёмная/светлая, состояния/ошибки, настенная панель, десктоп-модалка, компакт+idle, стадии «Открыть» + аудио)
**Референс:** `ru.inetra.intercom` (скриншоты экрана вызова с устройства) + `mockups/*.png`
**Статус:** ✅ реализовано (перевёрстка `frontend/`, pixel-perfect по `design.pen`; см. [plan-call-card-reverstka](plan-call-card-reverstka.md) + CHANGELOG `[Unreleased] › Changed`). Изначально — iteration 1 UX-контракт.

## 1. Краткое резюме решения

Карточка — это **машина состояний поверх одного видео-героя**. Единственный источник правды — `sensor.<intercom>_call_state` (`idle → ringing → connecting → active → ended/error`). Всё остальное (кнопки, статус, таймер, оверлеи) — производные от состояния.

Три опоры решения:

1. **Видео — главный и постоянный элемент.** Оно есть уже на входящем (до принятия), 16:9, доминирует во всех состояниях. Пользователь решает «открыть / говорить / игнор» по картинке, а не по тексту.
2. **«Открыть дверь» ≠ «Принять вызов».** Это два независимых действия в разных зонах, разных цветах и с разной защитой. Дверь можно открыть в любой момент, **не принимая звонок** — это ключевой сценарий домофона.
3. **Slide-to-open как защита от случайного нажатия.** Заимствовано из оригинального приложения (там это уже привычный жест «Сдвиньте, чтобы открыть»), адаптировано под HA: свайп на touch, hold-to-open на десктопе.

Микрофон и аудиовыход — **клиентские (браузерные) сущности, а не HA-entity** (см. §12): сервер HA не имеет доступа к микрофону/динамику смотрящего устройства. Карточка управляет ими локально через `getUserMedia` / `setSinkId`.

## 2. Анализ референса «Электронный Город»

Разобраны скриншоты экрана активного вызова (`Примерная 1 (п2)`) и главного экрана приложения.

**Визуальный язык:**
- Светлая тема, много воздуха, крупная типографика (заголовки ~28–34sp, «Вызов на домофон» — очень крупно).
- Акцент — зелёный (`#4CAF50`-семейство) для входа/принятия и приложения; красный (`#EF5350`) для отклонения/завершения; голубой ключ на слайдере.
- Скругления крупные (капсулы/pill у слайдера, круги 64–72dp у кнопок вызова).
- Иконки — тонкие линейные (телефон, микрофон, динамик, ключ, замок).

**Структура экрана вызова (сверху вниз):**
1. Header: домик-логотип + название точки доступа + «✕».
2. Видео 16:9 со скруглением и таймстампом-оверлеем.
3. Крупный **таймер по центру** (`00:17`) или заголовок «Вызов на домофон» до принятия.
4. **Slide-to-open** пилюля: ключ-thumb + «Открыть» + иконка замка; подпись «Сдвиньте, чтобы открыть». После свайпа → «Открыто» с зелёной подсветкой.
5. Нижний ряд круглых кнопок: `Выкл. микрофон` · `Завершить` (красный) · `Динамик`. До принятия — `Принять` (зелёный) · `Завершить`.

**Поведение слайдера (из серии кадров):** thumb едет слева-направо, при завершении фон становится зелёным и текст «Открыто», через пару секунд — сброс в исходное. Есть отдельный «bottom sheet» выбора аудио: `Без звука / Телефон / Динамик` с галочкой у выбранного.

**Паттерн навигации:** список точек доступа («Доступы») карточками с превью камеры и красной кнопкой-замком прямо на превью — быстрый «открыть без звонка» с главного экрана.

## 3. Сильные и слабые стороны оригинала

| Сильные стороны | Как переносим в HA |
|---|---|
| Видео — герой, видно до принятия | Сохраняем 1:1 |
| Slide-to-open защищает дверь | Сохраняем, добавляем hold-fallback для десктопа |
| Чёткое цветовое кодирование (зелёный/красный/акцент) | Мапим на токены темы HA (`--success/--error/--primary`) |
| Крупный таймер = «разговор идёт» | Сохраняем, статус-бейдж дополнительно |
| Быстрый «открыть» с превью на главном | Компактная Lovelace-карточка + idle-чипы точек |

| Слабые стороны оригинала | Наше улучшение |
|---|---|
| Нет явного состояния «соединение устанавливается» | Отдельный `connecting` со спиннером |
| Нет UX для «звук/микро недоступны» (нативное приложение имеет права) | Явные баннеры `mic_permission_required`, `audio_blocked`, `camera_unavailable` (браузерные ограничения) |
| Кнопка «Динамик» вводит в заблуждение в вебе (веб не всегда умеет менять sink) | Аудиовыход — **опционально**, скрываем если `setSinkId` не поддержан |
| Нет обратной связи по ошибке открытия | Явный временный `error` на слайдере + повтор |
| Автосброс/таймаут вызова не виден | Полоса окна ответа + счётчик `0:24` |
| Одна тема | Светлая + тёмная тема HA из коробки |

## 4. Пользовательские сценарии

1. **Входящий вызов.** FCM-push → `event.<intercom>_doorbell` (`ring`) → `call_state=ringing`. Карточка разворачивается (мобайл/панель — fullscreen-диалог; десктоп — модалка). Видео autoplay **muted**, статус «Входящий вызов», счётчик окна ~30с, слайдер «Открыть», кнопки `Отклонить`/`Принять`.
2. **Принимает вызов.** Тап `Принять` (=user gesture) → `answer` сервис → `connecting` (спиннер) → `active`. Тот же жест разблокирует autoplay-со-звуком: звук видео включаем, микрофон стартуем (если право есть). Кнопка `Принять` исчезает.
3. **Отклоняет вызов.** Тап `Отклонить` → `hangup` (CANCEL) → `ended` → авто-скрытие через ~2с → `idle`.
4. **Открывает дверь без ответа.** Свайп «Открыть» на `ringing` → `lock.unlock`. Звонок остаётся `ringing` (можно доснять/отклонить). Слайдер: `открываем → открыто`. Это главный «домофонный» сценарий.
5. **Открывает дверь во время разговора.** То же на `active`. Разговор не прерывается.
6. **Завершает разговор.** Тап `Завершить` → `hangup` (BYE) → `ended` → `idle`.
7. **Видео недоступно** (`camera_unavailable`). Фолбэк-плейсхолдер «Видео недоступно · аудиовызов продолжается»; аудио и открытие работают.
8. **Звук недоступен / autoplay заблокирован** (`audio_blocked`). Видео идёт muted. **Зона нажатия — всё видео целиком** (паттерн Telegram/Instagram/YouTube: тап в любом месте снимает mute); чип-CTA «Нажмите, чтобы включить звук» — визуальная подсказка-кнопка (с обводкой, внизу видео — не перекрывает лицо). Дублирующее явное действие — кнопка «Звук выкл.» (warning) в ряду.
9. **Микрофон недоступен / нет права** (`mic_permission_required`). Баннер «Нет доступа к микрофону · Вас не слышно» + кнопка «Разрешить». Downlink (слышно домофон) работает; открытие работает.
10. **Браузер блокирует autoplay со звуком.** = сценарий 8. Первый пользовательский тап по видео/CTA снимает mute.
11. **Браузер требует разрешение на микрофон.** Показываем баннер + запрос по кнопке (не авто-prompt, чтобы не пугать). После grant — мик стартует.
12. **Соединение устанавливается долго** (`connecting > 5с`). Спиннер + текст «Соединение…»; кнопка `Принять` → `Отменить`.
13. **Соединение оборвалось** (`error`/`connection_lost`). Красный статус, фолбэк «Соединение прервано · пробуем восстановить», кнопки `Повторить`/`Завершить`. Авто-скрытие error через 6с если не восстановилось.
14. **Дверь успешно открыта** (`door_opened`). Слайдер → зелёное «Открыто · 11:25», хаптик, авто-сброс через 3с.
15. **Ошибка открытия** (`door_error`). Слайдер → красное «Не удалось открыть · Повторить» (временно, 4с), затем возврат в покой.
16. **Несколько домофонов.** Активен один звонок за раз (как в приложении). Карточка привязывается к первому `ringing/active`. Список точек — в idle-чипах и в компактной карточке.
17. **Мобайл / панель / десктоп.** См. §10 — адаптив.

## 5. Состояния карточки

|---|---|---|---|---|---|---|
| `idle` | Плейсхолдер «Нет активного вызова» + чипы точек | Открыть любую дверь (чип) | — | «Видео появится при звонке» | — | Открытие — да (slide/hold) |
| `incoming_call` (ringing) | Видео muted, «Входящий вызов», окно 0:24 | Открыть, Принять, Отклонить | Микрофон, Аудио | «Сдвиньте, чтобы открыть» | Видео autoplay muted; отсчёт окна | Открытие — да |

| `opening_door` (слой) | Бегунок залип у правого края (оптимистично зелёный), `lock.unlock` в полёте | остальные действия | Слайдер | — | `lock.unlock` вызван | — |
| `door_opened` (слой) | Бегунок залип зелёным, текст «Открыто», подпись «Дверь открыта · время» | те же экрана | Слайдер (3с) | «Дверь открыта» | Хаптик; авто-сброс 3с | — |
| `door_error` (слой) | Бегунок вернулся в покой; под слайдером красная подпись «Не удалось открыть · Повторить» (4с) | Повторить | — | причина ошибки | Авто-сброс 4с | — |
| `call_ended` | Затемнённый кадр, «Вызов завершён 02:34» | Закрыть | все звонковые | — | Авто-скрытие ~2с → idle | — |
| `camera_unavailable` (слой) | Плейсхолдер video-off | звонковые/открытие | — | «Аудиовызов продолжается» | ретрай стрима | — |
| `audio_unavailable`/`audio_blocked` (слой) | CTA «Включить звук» на видео | Unmute по тапу | авто-unmute | «Нажмите, чтобы включить звук» | — | — |
| `microphone_permission_required` (слой) | warning-баннер + «Разрешить» | Запросить право | uplink | «Вас не слышно» | — | grant в браузере |
| `error` | Красный статус + фолбэк | Повторить, Завершить | — | текст ошибки | авто-скрытие 6с | — |

> `muted / speaker_select / opening_door / door_opened / door_error / camera_* / audio_* / mic_*` — это **слои поверх основного** `active`/`incoming`, а не отдельные экраны. Основное состояние = `sensor.*_call_state`; слои = локальный UI-стейт карточки.

│ │      [ видео звонящего ]   │   │
│ │  2026-06-25 18:42:07       │   │
│ └───────────────────────────┘   │
│                                 │

## 8. Логика кнопок

| Кнопка | Видна в | Действие | Цвет | Защита |
|---|---|---|---|---|
| **Открыть (slide)** | idle, ringing, connecting, active | `lock.unlock` (`button.*_open_door`) | `--primary` (акцент, НЕ красный) | swipe / hold |
| **Принять** | ringing | `elektronny_gorod.answer` | `--success` | нет (одиночный тап) |
| **Отклонить** | ringing, connecting | `elektronny_gorod.hangup` (CANCEL) | `--error` | нет |
| **Завершить** | active, error | `elektronny_gorod.hangup` (BYE) | `--error` | нет |
| **Микрофон** | active | локально mute/unmute uplink | нейтр./`--error` when off | нет |
| **Звук** | active | локально mute/unmute входящего аудио (`video.muted`) | нейтр./`--warning` when off | нет |
| **Повторить** | error/connection_lost/door_error | retry (answer/open) | `--primary` | нет |
| **Закрыть (✕)** | все | свернуть карточку (звонок продолжается на фоне) | нейтральный | нет |

**Принципы:** `Открыть` — единственная кнопка с защитой (см. §9). Деструктив (`Отклонить/Завершить`) — красный, но одиночный тап (быстрое действие важнее). `Принять` — зелёный и крупнее. `✕` не завершает звонок — только сворачивает.

## 9. UX безопасности — открытие двери

**Риск:** случайно открыть дверь = впустить чужого. Кнопка должна быть заметной, но не «нажимаемой мимоходом».

**Три варианта:**

1. **Обычная кнопка + модальное подтверждение.** Плюсы: просто, доступно. Минусы: два тапа, диалог перекрывает видео, «замыленный да» → защита слабеет. Хорошо для скринридеров.
2. **Long-press (hold, 600–800мс) с кольцом-прогрессом.** Плюсы: один непрерывный жест, наглядный прогресс, отменяемо (отпустил — отмена). Минусы: hold плохо работает с мышью на десктопе (зажать кнопку неудобно), хуже для моторики.
3. **Slide-to-open (свайп thumb по пилюле).** Плюсы: «намеренный» жест как в оригинальном приложении → нулевое переобучение; невозможно случайно; естественно на тач; прогресс виден. Минусы: требует ширины; на десктоп-мыши свайп неудобен.

**Выбор для HA: slide-to-open на touch + hold-to-open на desktop (адаптивно по `pointer: coarse/fine`), с подтверждением-fallback для screen-reader.**

Почему: (а) 1:1 привычка из «Электронного Города» — пользователь уже знает «Сдвиньте, чтобы открыть»; (б) физически исключает случайное открытие; (в) HA-панели и телефоны — touch, десктоп — редкий кейс (там hold или confirm). `open_action: auto|slide|hold|tap` — конфиг для ручного override.

**Габариты (design best practices):** минимальная цель нажатия — 44pt (Apple HIG) / 48dp (Material); подушечка большого пальца ≈ 10–12 мм. Для главного действия берём с запасом: **бегунок 68pt, трек 80pt** — слайдер выше кнопок ряда (68pt) и визуально доминирует как главное действие. Длина трека — **~300pt, не на всю ширину**: свайп — формальное подтверждение намерения, а не «время подумать»; короткий ход быстрее и удобнее одной рукой. Все круглые кнопки в ряду — **одного размера** (равнозначные действия, без «акцентной» кнопки).

**Стадии слайдера (упрощённая модель, см. секцию 06 макетов):**
Метафора: ключ 🔑 едет от закрытого замка 🔒 (слева) к открытому 🔓 (справа).
1. **Покой** — бегунок-ключ слева (акцент `--primary`), по центру «Открыть», справа 🔓. Закрытый 🔒 лежит **под бегунком** и в покое не виден.
2. **Тянем** — ключ уезжает вправо и **перестаёт перекрывать** 🔒 — тот просто проявляется у левого края (без анимаций, чистое перекрытие слоёв). За бегунком приглушённый шлейф (`--primary` @ ~15% прозрачности).
3. **Довели до края** — бегунок **залипает** справа, становится **зелёным**, текст меняется на «Открыто»; 🔒 остаётся виден слева (хаптик; авто-сброс через 3с).

Ошибка открытия — **не** отдельное состояние слайдера: бегунок возвращается в покой, под слайдером подпись `--error` «Не удалось открыть · Повторить» (4с).

**Hold-to-open (десктоп/мышь) — зеркально слайдеру, с видимым прогрессом (см. секция 06 макетов).** Ключевой принцип: **кнопка не меняет содержимое** — иконка 🔑 и надпись «Удерживайте, чтобы открыть» остаются на месте и по центру; ключ **не едет**, текст **не подменяется**. Меняется только заливка-прогресс под контентом.
1. **Покой** — пилюля `--primary` (обводка), 🔑 + «Удерживайте, чтобы открыть».
2. **Удерживаем** — под неизменным контентом бежит заливка `--primary` (полупрозрачная) слева направо (~0.8с) = «сколько ещё держать».
3. **Отпустили раньше** — заливка **откатывается назад** к нулю, ничего не происходит (отменяемо); контент тот же.
4. **Успех** — пилюля зелёная «Открыто» (хаптик; авто-сброс 3с).
5. **Ошибка** — пилюля в покой + подпись `--error` «Не удалось открыть · Повторить» (4с).

Прогресс обязателен: без него hold — «угадайка по времени». Длительность ~0.8с (≈slide). Реализация: заливка — `::before`/overlay с `width: 0→100%` (`transition` на `pointerdown`, reverse на раннем `pointerup`); контент кнопки статичен, цвет текста читаем и на заливке, и на фоне.

**Отличие «Открыть» от «Принять»:** разные зоны (открыть — над кнопками вызова), разные цвета (`--primary` vs `--success`), разные интеракции (жест vs тап), разные иконки (🔑 vs 📞). Открытие доступно всегда, приём — только на `ringing`.

## 10. Адаптивность

**Мобайл (≤480px)** — вертикаль, видео 16:9 на всю ширину, крупные круглые кнопки (64/72pt), действия в нижней трети, одна рука. При входящем — fullscreen-диалог поверх дашборда. *(секция 01/03 макетов)*

**Планшет (≤1024px, landscape)** — видео крупное; действия можно сбоку (2 колонки) либо снизу; текст/кнопки читаемы с ~1м.

**Настенная панель (fullscreen)** — полноширинная шапка сверху (имя+адрес слева, статус+автосброс справа); ниже видео-герой **строго 16:9 без кропа** (нативное соотношение стрима домофона), максимизирован за счёт узкой колонки контролов (~340pt); колонка контролов = высоте видео: слайдер по вертикальному центру видео, кнопки (равные, 100pt) по нижней кромке; остаток высоты — спокойный нижний отступ панели, а не разрыв между шапкой и видео. Минимум мелочей. Fullscreen-разворот по `ring` (kiosk). *(секция 04 макетов)*

**Десктоп (≥1024px)** — компактная карточка в дашборде; при входящем — **модалка по центру** с тенью, hold-to-open, кнопки в строку. Действия без скролла. *(секция 05 макетов)*

**Компактный Lovelace** — одна строка: мини-превью (96×72, LIVE) + имя + статус/таймер + 3 быстрые кнопки (🔑/❌/📞). Для дашборда с другими карточками. *(секция 05 макетов)*

**Fullscreen** — видео максимум, действия оверлеем снизу на scrim-градиенте; кнопки/оверлеи **не перекрывают центр кадра** (лицо звонящего). Статус/таймер — верх, действия — низ.

Реализация — `container-queries` по ширине карточки (не медиа-запросы окна), чтобы работало в любой ячейке grid.

## 11. Визуальная иерархия

1. **Видео** — главный элемент, максимальный вес, 16:9, всегда вверху/в центре.
2. **Имя домофона + статус** — сразу над видео (кто и что).
3. **«Открыть»** — заметная зона под видео, акцентный цвет, но защищённый жест.
4. **Приём/отклонение** — нижняя треть, самые крупные круги, зелёный/красный.
5. **Микрофон/аудио** — вторичные, нейтральные круги, меньше визуального веса.

Разделение основное/дополнительное: primary-действия — заполненные крупные круги/пилюли; дополнительные (микрофон, аудио, закрыть) — нейтральный `--elevated` фон. Деструктив — только красный. Успех — только зелёный. Акцент двери — `--primary` (никогда не красный, чтобы не путать с «завершить»).

## 13. Технические ограничения (WebRTC / autoplay / mic / audio-output)

**Autoplay звука.** Браузеры блокируют autoplay **со звуком** до пользовательского жеста; muted-autoplay разрешён. → Видео на `ringing` играет **muted**. Принятие (`Принять`) — это и есть жест → сразу после него `video.muted=false` разблокирует звук легально. **Edge-case:** авто-разворот fullscreen на панели без тапа → состояние `audio_blocked`: чип-CTA «Нажмите, чтобы включить звук» внизу видео, при этом **кликабельно всё видео** (чип — только подсказка; тап-зона большая, промахнуться нельзя), дубль — кнопка «Звук» в ряду действий. iOS Safari строже — всегда закладываемся на этот механизм.

**Микрофон (`getUserMedia`).** Требует **secure context** (HTTPS или `localhost`) + разрешение. **HA по HTTP на LAN-IP → `getUserMedia` недоступен** (не secure origin) → двусторонний звук невозможен. Обязательно документировать: **uplink требует HTTPS** (или запуск как installed PWA / trusted origin). Если `navigator.permissions.query({name:'microphone'})` = `granted` **и** secure context → mic autostart на `active`. Иначе — баннер «Разрешить микрофон», prompt **по кнопке** (не авто, чтобы жест был осознанным). Downlink (слышимость домофона) от прав микрофона не зависит.

**Выбор аудиовыхода — не делаем.** Проверено: в Home Assistant (custom card в браузере/Companion WebView) надёжный выбор устройства вывода невозможен — `setSinkId` есть только в Chromium desktop/Android, требует mic-права для лейблов, отсутствует на iOS Safari и в WebView ведёт себя непредсказуемо. Решение: **никакого `speaker_select`** — вместо него кнопка «Звук» (простой mute/unmute входящего аудио). Маршрутизация динамик/наушник/BT остаётся за ОС.

**WebRTC / stream.** Downlink = `camera.<intercom>_call` через **go2rtc** (HA-native WebRTC, 4G-safe, без внешней экспозиции). Uplink (микрофон) = отдельный WS-канал `elektronny_gorod/intercom_uplink` (браузер → HA WS → SIP RTP). Peacetime-видео (без звука) — обычный `camera.*_video`.

**Контракт плеера (тап по видео).** Рендер — `ha-camera-stream` (HA-native; внутри `<video>` в shadow DOM), fallback `webrtc-camera`. Требования, чтобы UX «тап по видео = включить звук» работал:
1. **Без `controls`** — иначе в Chrome/Firefox клик по видео = play/pause (пауза живого звонка!) и нативная панель дублирует нашу кнопку «Звук». Видео — chromeless-поверхность; весь UI — наш. ⚠️ В текущем `call-video.ts:102` `controls` передан — убрать.
2. **Прозрачный tap-слой поверх плеера** (`inset:0`) в карточке — единственный обработчик тапов по видео (плеер указатель не видит); в `audio_blocked` снимает mute.
3. **Unmute синхронно в обработчике жеста** (`muted=false` в click) — autoplay-политика удовлетворена, `.muted` реактивно доходит до `<video>` без прерывания потока.
4. Fallback-нюанс: `webrtc-camera` при смене `muted` сейчас пересоздаётся (re-negotiation ~1–2с) — либо мутить его `<video>` напрямую, либо принять как ограничение fallback.

**Что можно в custom Lovelace card:** всё UI, slide/hold-жест, unmute-on-gesture, `getUserMedia`+VU-метр, таймер, оркестрация состояний, `setSinkId`. **Что требует отдельного (уже есть):** `mic-controller.ts`, аудио-бридж на стороне HA (go2rtc + RTP latching), WS-uplink endpoint. Нативные HA-карты (picture-glance) **не** дадут слайдер/микрофон → нужна кастомная карта (есть: `eg-intercom-call-card`).

## 14. Home Assistant entities / services

**Важная корректировка базового варианта из ТЗ:** микрофон и аудиовыход — **клиентские браузерные сущности, а не HA-entity**. `switch.*_microphone` и `select.*_audio_output` **не могут существовать** корректно: сервер HA не имеет доступа к микрофону/динамику *смотрящего браузера* (у разных зрителей — разные устройства). Ими управляет карта локально. Не моделируем их entity.

**Обязательные:**
- `sensor.<intercom>_call_state` — enum `idle|ringing|connecting|active|ended|error`; атрибуты `call_id, intercom_name, started_at, access_control_id, place_id`. **Единый источник правды.**
- `camera.<intercom>_call` — видео+аудио downlink во время звонка (go2rtc). Недоступна вне звонка.
- `event.<intercom>_doorbell` — события `ring`/`ended` (триггер разворота карты/уведомления).
- `lock.<intercom>` (или `button.<intercom>_open_door`) — открыть дверь → `/accessControlOpen`.
- Сервисы: `elektronny_gorod.answer`, `elektronny_gorod.hangup`.

**Опциональные / расширенный UX:**
- `camera.<intercom>_video` — peacetime-превью (для idle/компакта/чипов).
- Длительность звонка — **вычислять в карте** из `started_at`, не заводить `sensor.*_call_duration` (лишний entity, дёргается ежесекундно).
- `binary_sensor.<intercom>_call` — избыточен при наличии `call_state`; оставить только если нужен простой триггер для DIY-автоматизаций.

**Через сервисы, а не entity:** `answer/hangup/open` — действия, им не нужны отдельные `button.*_accept/reject/end` entity (кнопки-entity плодят сущности; карта зовёт сервисы напрямую). `button.*_open_door` оставить как удобную DIY-точку.

**Uplink микрофона:** WS-команда `elektronny_gorod/intercom_uplink`, не entity.

## 16. Acceptance criteria

- [ ] Видео звонящего видно **до** принятия вызова (`ringing`).
- [ ] Видно название домофона и (опц.) адрес.
- [ ] Пользователь может принять вызов (`answer`).
- [ ] Пользователь может отклонить вызов (`hangup/CANCEL`).
- [ ] Пользователь может открыть дверь **без** принятия (`ringing` → `lock.unlock`).
- [ ] Пользователь может открыть дверь во время разговора (`active`).
- [ ] После принятия кнопка «Принять» исчезает; появляется `Микрофон/Динамик/Завершить`.
- [ ] На экране разговора виден активный статус (зелёный) и таймер.
- [ ] Микрофон стартует по умолчанию, если право уже выдано **и** secure context.
- [ ] Если микрофон недоступен/нет права — понятный баннер + кнопка «Разрешить»; downlink работает.
- [ ] Если звук нельзя включить автоматически — CTA «Включить звук»; первый тап снимает mute.
- [ ] Кнопка открытия защищена от случайного нажатия (slide/hold).
- [ ] Слайдер — самый крупный контрол экрана (трек 80pt, бегунок 68pt ≥ HIG 44 / Material 48); длина ~300pt, не на всю ширину.
- [ ] Все круглые кнопки в одном ряду — одинакового размера.
- [ ] Стадии открытия видны: тянем → открываем → **открыто** (успех) / **ошибка**.
- [ ] Ошибка открытия показывается пользователю (временный красный + повтор).
- [ ] Корректно работает на мобайл, десктоп, планшет, панель, компакт, fullscreen.
- [ ] Поддержаны светлая и тёмная темы HA (только токены).
- [ ] Элементы управления не перекрывают лицо звонящего / центр кадра.
- [ ] Интерфейс понятен без инструкции; `✕` сворачивает, но не завершает звонок.
- [ ] `connecting` показывает прогресс; долгое соединение не выглядит как зависание.
- [ ] `camera_unavailable` не ломает звонок (аудио + открытие работают).

## 17. Открытые вопросы к разработчику интеграции

1. **HTTPS для uplink.** Подтвердить: у целевых установок HA доступен по HTTPS/`localhost`? Иначе двусторонний звук физически невозможен (`getUserMedia`) — нужен ли fallback «только слушать + открыть»?
2. **Окно ответа.** `CallInvalidated` даёт точный дедлайн (~30с) всегда, или иногда только fallback 35с? От этого — показывать точный счётчик или «до ~30с».
3. ~~**Аудиовыход.**~~ **Решено:** выбор устройства вывода в HA невозможен — убран; кнопка «Звук» = mute/unmute.
4. **Открыть на `idle`/из компакта.** Разрешаем открывать дверь без входящего звонка (просто «пустить курьера»)? Тогда idle-чипы = slide-confirm.
5. **Несколько одновременных звонков.** Гарантированно один за раз, или возможна очередь (2 домофона звонят)? Нужен ли UI переключения/очереди сверх «активен один»?
6. **`ended` vs авто-скрытие.** Сколько держать экран `call_ended` перед сворачиванием (2с ок?) и нужен ли лог «последний вызов».
7. **Хаптик/звук рингтона** на панели/мобайле — на стороне карты (Web Vibration/Audio) или Companion-app уведомления?
8. **Идентичность точек.** `intercom_name` из атрибута достаточно, или нужен маппинг на человекочитаемые имена/адрес в конфиге карты?
9. **Мик-VU-метр.** Показывать индикатор уровня микрофона (что «вас слышно»)? Полезно, но требует AudioContext-анализа.
10. ~~**Кнопка «Динамик» на активном.**~~ **Решено:** заменена кнопкой «Звук» (mute/unmute), вопрос снят.
