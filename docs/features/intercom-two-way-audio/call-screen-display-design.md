# Дизайн: показ экрана вызова (video + downlink-audio) через HA-native

- **Date:** 2026-06-23
- **Status:** approved (brainstorming) → к плану реализации
- **Связь:** [audio-bridge-design.md](audio-bridge-design.md), [call-answer-model.md](call-answer-model.md),
  ADR-0012 (register-on-ring), audit A-84 (go2rtc config bloat).

## Goal

Показать экран вызова домофона в HA-дашборде — **видео + звук гостя инлайн**
(открыл экран → сразу видео со звуком), с заделом под **микрофон** (two-way) —
так, чтобы работало **на 4G** (удалённо), с **одним** go2rtc-инстансом, без
экспозиции go2rtc наружу.

## Контекст / тупик, из которого вышли

Сессия 2026-06-23 перебрала пути показа и уперлась в комбинацию ограничений:
- **webrtc-camera (AlexxIT) в режиме `url:`** играет инлайн-звук, но на 4G требует
  **прямого** доступа к go2rtc (экспозиция) и не принимает auth отдельным полем;
  плюс при прямом потреблении **камерных** стримов ловит **EOF** (одноразовый
  operator-RTSP протухает; рефреш только через HA `stream_source`).
- **Два инстанса go2rtc** (отдельный публичный) — антипаттерн для пользователей
  интеграции, отвергнут.
- **Парадигма пользователя:** go2rtc либо внешний+пароль, либо LAN; «HA проксирует
  авторизованным».

**Разблокировка (live-подтверждено):** HA-native показ go2rtc-камеры **играет
звук** (в полноэкранном плеере, на 4G). Значит WebRTC-плеер HA со звуком на 4G
работает — звук вызова можно отдать **через HA-native**, без экспозиции.

## Архитектура

Интеграция заводит **одну** camera-сущность `camera.intercom_call`; HA-native
отдаёт её по WebRTC (видео+звук). go2rtc остаётся в LAN, креды держит HA,
фронтенд ходит за HA-логином, на 4G — через авторизованный канал HA (как
фуллскрин-показ камер, который уже играет звук).

```
SIP downlink (мост) ─┐
                     ├─→ go2rtc 1984 стрим eg_intercom_call ─→ camera.intercom_call
видео домофона (свежий)┘     (LAN)                                  │
                                                                    ↓ HA-native WebRTC (за HA-логином)
                                                  дашборд: инлайн video+audio (4G ok)
```

### Сущность `camera.intercom_call`

- `Camera` + `CameraEntityFeature.STREAM`; `unique_id = {DOMAIN}_{entry_id}_intercom_call`
  (scoped к entry — multi-entry-safe); `_attr_name = None` (имя из `device_info`,
  как `ElektronnyGorodCamera`). Реальный `entity_id` (single-entry): `camera.vyzov_domofona_intercom_call`.
- `stream_source()`:
  - **активный вызов** → пересобрать `eg_intercom_call` со **свежим** operator-URL
    видео домофона + аудио-мост, upsert в go2rtc, вернуть
    `rtsp://<rtsp_host>:8554/eg_intercom_call`. Свежий video-URL берём **тем же
    путём, что `camera.py`** — через operator-API/coordinator (а не правкой
    `camera.py`); если для DRY нужен общий хелпер — это **низкорисковый extract**
    функции получения source (решает план), без изменения stream-lifecycle камеры;
  - **нет вызова** → `None` (сущность недоступна; экран её скрывает).
- 🔑 **Рефреш-на-открытии убирает EOF (A-84-смежно):** `stream_source()` зовётся
  HA при каждом открытии плеера → каждый раз свежий источник, как у рабочих камер.
- 🔴 stream-lifecycle камеры (`stream_source`/recovery hot-path) **не трогаем** —
  отдельная сущность, нулевой риск «мигания» (A-66/A-71). Допустим лишь
  низкорисковый **extract** функции получения source для DRY (если план решит).

### Граница (изоляция)

`camera.intercom_call` ↔ `DoorbellCallController` через узкий интерфейс: «активный
вызов — какая камера (`access_control_id`→camera) + адрес моста». Контроллер это
уже знает. Сущность не лезет в SIP; контроллер не лезет в camera-rendering.

## Forward-compat: микрофон (uplink, Slice 2)

camera-сущность — правильный фундамент: у HA **нативный two-way audio** (камера
объявляет поддержку → во встроенном плеере кнопка микрофона; go2rtc штатно держит
WebRTC-backchannel). Путь uplink (следующий слайс):

```
микрофон браузера → HA WebRTC (тот же канал) → go2rtc backchannel
   → мост (обратная ветка) → SipManager.uplink_provider (сейчас тишина) → RTP в домофон
```

🔴 **Риск §6.5** (маршрутизация backchannel → SIP — exec/RTP/обратный HTTP)
остаётся → uplink отдельным слайсом со своим PoC. Но направление выбрано верно:
camera + HA-native two-way чище, чем two-way хаком в карточке.

## Показ инлайн со звуком (единственный verify-пункт)

Нужна карточка, встраивающая **нативный WebRTC-плеер HA** инлайн (тот, что играет
звук в фуллскрине). `picture-entity` не годится (инлайн глушит звук).

Порядок проверки:
1. **`webrtc-camera` в режиме `entity:`** (`entity: camera.intercom_call`, не `url:`)
   — уже установлена, умеет инлайн-звук; в `entity`-режиме сигналинг через HA →
   шанс, что на 4G заведётся (в отличие от `url:`). Если да — ничего не ставим.
2. **`advanced-camera-card`** (`live_provider: ha`) — специально встраивает нативный
   HA-плеер инлайн со звуком, на 4G надёжно. Минус — поставить карточку (HACS).
3. **Fallback** — видео инлайн + звук по тапу в фуллскрин (нежелательно, только
   если 1–2 не дадут инлайн).

Сущность (Архитектура) твёрдая в любом случае; меняется лишь карточка.

### Результат (live-проверено 2026-06-23)

✅ **Вариант 1 сработал:** `webrtc-camera` в режиме `entity: camera.vyzov_domofona_intercom_call`
(`muted: false`) даёт **видео+звук инлайн на 4G**, без экспозиции go2rtc, без EOF (рефреш-
на-открытии). Дашборд: ringing → `picture-entity` (видео домофона), in-call → одна
`webrtc-camera`-карточка. `advanced-camera-card` (вариант 2) не понадобился.

⚠️ **Авто-unmute:** звук стартует выключенным — **политика автоплея браузера** блокирует
автозапуск со звуком (`muted: false` выставлен, но браузер глушит до взаимодействия).
Надёжно: один тап по плееру (браузер запоминает на сессию). Полный авто-звук политикой
браузера не гарантируется.

## Что решает по болям

| Боль | Решение |
|---|---|
| auth в webrtc-camera | не нужен — креды go2rtc у HA, фронтенд за HA-логином |
| EOF (протухший operator-URL) | рефреш-на-открытии `stream_source()` |
| 4G без экспозиции | HA-native туннель (как фуллскрин камер) |
| два инстанса | один go2rtc 1984 |
| микрофон позже | HA-native two-way + go2rtc backchannel (Slice 2) |

## Тестирование

- **Юнит (TDD):** `camera.intercom_call.stream_source()` — активный вызов → свежий
  combined-стрим + RTSP; нет вызова → `None`; рефреш-на-открытии → пересборка
  источника (mocked go2rtc/controller).
- **Live (прод):** звонок → Ответить → на 4G видео+звук инлайн, без EOF; повторный
  вход работает; go2rtc наружу не торчит.
- **Регресс:** `camera.py` не тронут → существующие камеры без изменений (полный suite).

## Отвергнутые альтернативы

- **webrtc-camera `url:` + внешний+пароль go2rtc** — экспозиция, auth не вводится,
  EOF на камерных стримах. (Подход 3 brainstorm.)
- **Подмешать аудио в существующую camera-сущность** — правка go2rtc-источника
  `camera.py` во время вызова → риск «мигания» A-66/A-71. (Подход 2 brainstorm.)
- **Два инстанса go2rtc (публичный отдельный)** — антипаттерн для пользователей.
