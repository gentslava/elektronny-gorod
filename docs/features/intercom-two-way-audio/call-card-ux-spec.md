## 1. Краткое резюме решения

Делаем **frontend-расширение с двумя путями использования** (по решению владельца):

1. **Toolkit** — интеграция отдаёт полный набор сущностей/сервисов (`camera.*`,
   `event.*`, `lock.*`, `sensor.*_call_state`, сервисы `answer`/`hangup`,
   WS-uplink микрофона). На них пользователь собирает **свой** экран на любых
   карточках (стоковые HA, `webrtc-camera`, `advanced-camera-card`, своя карточка).
2. **Из коробки** — наша кастомная карточка **`eg-intercom-call-card`** (Lit + TS),
   которая авторизуется через HA, подключается одной строкой и **сразу выглядит
   и работает хорошо**, повторяя привычный по приложению UX, но в **родном
   HA-облике** (наследует тему пользователя, M3-компоненты, без Android-идиом).

**Два главных состояния** — `incoming_call` (входящий) и `active_call` (разговор) —
плюс полный набор промежуточных/ошибочных. **Видео — главный элемент**. Открытие
двери — **независимо** от приёма вызова, с защитой от случайного нажатия двумя
вариантами: **нативный (tap/hold)** для DIY/стока и **Slide-to-open** в нашей
карточке — адаптивно: slide на тач, hold/tap на десктопе (slide мышью неудобен).

**Жёсткие ограничения платформы, которые формируют дизайн** (не «хотелки», а закон):
- **Автозвук со звуком в браузере блокируется** → звук гостя стартует muted, нужен
  один жест («оверлей-кнопка Включить звук»). Гарантировать авто-звук нельзя.
- **Микрофон требует HTTPS-origin + разрешение** (`getUserMedia`) → «по умолчанию
  вкл» только если разрешение уже выдано и origin secure; иначе — CTA «Разрешить».
- **Аудиовыход «в телефонный динамик» (earpiece) недоступен** веб-странице →
  bottom-sheet «Динамик/Телефон/Без звука» из приложения **не воспроизводим**;
  оставляем только **Mute** + системная громкость.
- **Окно ответа ~30с** (`CallInvalidated`, домофон сам сбрасывает) → карточка
  должна авто-истекать.
- Видео живёт через **HA-native WebRTC** (go2rtc в LAN, за HA-логином, 4G ok без
  экспозиции); EOF снят рефрешем-на-открытии (см. call-screen-display-design.md).

## 2. Анализ референса приложения «Электронный Город» (`ru.inetra.intercom`)

Изучено вживую через ADB (свежие скриншоты 2026-06-24): главный экран, экран
вызова (полный цикл), разговор, экран событий, состояние загрузки.

### 2.1 Визуальный язык (эмпирически)

| Аспект | Приложение |
|---|---|
| Тема | Светлая, белый фон, очень много воздуха |
| Акцент | Мягкий **салатовый** (фон pill-ов и чипов), логотип — зелёный |
| Текст | Почти-чёрный navy для заголовков (крупный жирный гротеск), серый body |
| Форма | Скруглено всё: карточки ~16–20px, pills/чипы — полностью круглые |
| **Красный** | `#FF3B30`-подобный: и «Открыть дверь», и «Завершить» (⚠️ два смысла) |
| **Зелёный** | «Принять» (вызов) + завершённое «Открыто» |
| Иконки | Линейные, в кружках; mdi-подобные |
| Навигация | Нижний таб-бар (Мой дом / События / Люди / Кабинет), active = жирный чёрный |
| Загрузка | **Skeleton-shimmer** (плейсхолдеры-плашки) |
| Списки | Группировка по датам, фильтр-чипы (Дата/Устройства/Люди) |

### 2.2 Экран вызова — поток (ключевой референс)

1. **Входящий** (`Вызов на домофон`): шапка `[лого] Название домофона [✕]` → **видео
   гостя** (скруглённое, вверху) → большой заголовок «Вызов на домофон» → **слайдер
   `(🔑)──► Открыть 🔓` с подписью «Сдвиньте, чтобы открыть»** → внизу две круглые
   кнопки `(Принять, зелёная)` `(Завершить, красная)`.
2. **Разговор**: то же, но вместо заголовка — **таймер `00:05`**; внизу три круглые
   кнопки `(Выкл. микрофон)` `(Завершить, красная)` `(Динамик)`. Кнопки «Принять» нет.
3. **Аудиовыход** (тап по «Динамик»): **bottom-sheet** со скримом — `Без звука` /
   `Телефон` / `Динамик (✓)`.
4. **Открыто**: слайдер доехал — `🔒 Открыто (🔑)` зелёным; нижние кнопки те же.

### 2.3 Открытие двери в приложении

Везде (главный экран, экран вызова) открытие = **красная круглая кнопка-замок** или
**slide-to-open**. На главном экране у каждой камеры — красный кружок-замок. Дверь
открывается **без** разговора.

## 3. Сильные и слабые стороны оригинала

**Сильные (заимствуем смысл, не форму):**
- Видео — доминанта, сразу видно гостя до решения.
- Открытие двери **отделено** от приёма вызова и доступно сразу.
- Slide-to-open — защита от случайного открытия + приятный, понятный жест.
- Чёткое различие состояний: входящий (Принять/Завершить) vs разговор (Микрофон/
  Аудио/Завершить + таймер).
- Чистая иерархия, крупные тач-цели, управление одной рукой.

**Слабые стороны → конкретные решения (что делаем вместо):**

| Слабость оригинала | На что исправляем (решение) |
|---|---|
| 🔴 Два разных «красных» (Открыть и Завершить) — семантический конфликт | Красный (`--error-color`) — **только** «Завершить». «Открыть» = `--primary-color` (accent) для слайдера/кнопки; завершённое «Открыто» = краткий `--success-color` + `mdi:lock-open-check`; ошибка = краткий `--error-color` + `mdi:lock-alert` (transient, не resting-цвет). «Принять» = `--success-color`. Различие Открыть↔Принять усилено **формой**: Открыть = слайдер/hold, Принять = кнопка |
| Bottom-sheet аудиовыхода с «Телефон» (earpiece) — в вебе невоспроизводимо | Убираем sheet целиком → **один тогл Mute/Unmute** (`mdi:volume-high`/`mdi:volume-off`); громкость системная. «Телефон/Динамик» не показываем |
| Android-идиомы (круглый FAB с подписью снизу, bottom-sheet, slide-to-unlock) — инородны в HA | Транслируем в M3/HA: действия — `ha-control-button`/`tile`/`button`; вместо bottom-sheet — `more-info`-диалог/inline; slide — на стиле `ha-control-slider` + адаптив (тач→slide, десктоп→hold/tap). Только theme-токены и `mdi:` |

**⏳ Реально открытый пункт (решаем live, НЕ зафиксировано):**
- **Индикатор окна 30с / таймер разговора.** В приложении — секундомер (счёт вверх),
  явного индикатора окна нет. По умолчанию: карточка **авто-скрывается** по истечении
  окна; таймер — конфигурируемый и ненавязчивый/скрытый. Показывать ли обратный
  отсчёт окна — **решаем после live-проверки** (владелец: «надо посмотреть, не будет
  ли напрягать»). См. §13, §17 п.6.

**Не портируем (не слабость, а сознательный отказ от копирования):**
- Skeleton-загрузка приложения → нативные HA-индикаторы (спиннер/состояния `hass`),
  без воспроизведения shimmer.

## 4. Пользовательские сценарии

| # | Сценарий | Поведение (HA) |
|---|---|---|
| 1 | Входящий вызов | `event` ring → `sensor.*_call_state=ringing`; карточка/уведомление показывают видео домофона (без звука) + Принять/Отклонить/Открыть; таймер окна 30с |
| 2 | Принять | Тап «Принять» → `elektronny_gorod.answer` → REGISTER→INVITE→200 OK (<1с, `connecting`) → `active`; первый жест снимает mute → слышно гостя; (микрофон по политике — см. §13) |
| 3 | Отклонить | Тап «Отклонить» → `elektronny_gorod.hangup` (до ответа = decline/CANCEL) → `ended` → карточка скрывается |
| 4 | Открыть без ответа | Slide/hold/confirm «Открыть» → `lock.unlock` (accessControlOpen), **не трогая SIP**; вызов продолжает звонить, можно затем Принять или Отклонить |
| 5 | Открыть во время разговора | То же «Открыть» поверх активного видео; разговор не прерывается; overlay «Открыто» |
| 6 | Завершить разговор | Тап «Завершить» → `elektronny_gorod.hangup` (BYE) → `ended` → скрытие; микрофон/плеер останавливаются |
| 7 | Видео недоступно | Стрим EOF/ошибка → `camera_unavailable`: плейсхолдер `mdi:cctv-off` + «Видео недоступно» + Повторить; **аудио и кнопки работают** |
| 8 | Звук недоступен | downlink 0 / WebRTC-audio fail → `audio_unavailable`: видео есть, бейдж «Звук недоступен» |
| 9 | Микрофон недоступен | Нет устройства/`getUserMedia` fail → кнопка микрофона disabled + подсказка |
| 10 | Браузер блокирует автозвук | Старт muted → **оверлей «🔊 Включить звук»** поверх видео; тап включает (браузер помнит на сессию) |
| 11 | Браузер требует разрешение микрофона | `microphone_permission_required`: кнопка «🎤 Разрешить микрофон» → `getUserMedia` prompt; до выдачи микрофон выкл |
| 12 | Соединение устанавливается долго | `connecting`: спиннер на кнопке + «Соединение…»; таймаут → `error` + Повторить |
| 13 | Соединение оборвалось | SIP BYE/таймаут/сеть → `ended`/`error`: «Вызов завершён»/«Связь потеряна» + (если уместно) Перезвонить недоступен (домофон инициатор) |
| 14 | Дверь открыта | `lock` open → overlay-успех `mdi:lock-open-check` «Открыто» (2–3с) → возврат |
| 15 | Ошибка открытия | `lock.unlock` исключение/таймаут → overlay-ошибка `mdi:lock-alert` «Не удалось открыть» + Повторить |
| 16 | Несколько домофонов | Один активный звонок одновременно (как в приложении); карточка биндится к активному, имя резолвится в заголовок; остальные домофоны — обычные камеры |
| 17 | Вызов на мобильном | Companion **actionable-уведомление** (Ответить/Сбросить/Открыть, blueprints) ИЛИ карточка в мобильном вебе HA; вертикальный layout |
| 18 | Вызов на настенной панели | Always-on dashboard; при звонке — fullscreen (авто-навигация на view или `browser_mod.popup`); крупные кнопки |
| 19 | Вызов на десктопе | Компактная карточка на дашборде; при звонке — модалка (`browser_mod`) или подсветка карточки; всё без прокрутки |

## 5. Состояния карточки

`sensor.<intercom>_call_state` (NEW, см. §14) — единый источник истины: enum
`idle | ringing | connecting | active | ended | error` + атрибуты
(`call_id`, `intercom_name`, `camera_entity`, `lock_entity`, `started_at`).
Door-состояния и audio/mic — **ортогональные слои** поверх основного.

| Состояние | Откуда | Видит пользователь | Доступно | Заблокировано | Авто-действие |
|---|---|---|---|---|---|
| `idle` | `call_state=idle` (camera.intercom_call unavailable) | Карточка скрыта или мини-превью домофона + «Нет вызова» | — | все call-кнопки | — |
| `incoming_call` | `call_state=ringing` | Видео домофона (без звука), заголовок «Входящий вызов», имя, таймер окна | Принять, Отклонить, **Открыть** | Микрофон, Аудио, Завершить | старт обратного отсчёта 30с |
| `connecting` | `call_state=connecting` (после `answer`) | Видео + спиннер «Соединение…» | Отклонить/Отмена, **Открыть** | Принять, Микрофон | переход в `active` при 200 OK; таймаут→`error` |
| `active_call` | `call_state=active` (camera.intercom_call available) | Видео+звук гостя, таймер | **Открыть**, Микрофон, Mute/Unmute, Завершить | Принять | первый жест → unmute; **микрофон авто-старт, если permission granted** (иначе CTA «Разрешить») |
| `muted` (слой) | card-local: mic off | Иконка микрофона перечёркнута + «Микрофон выкл» | toggle микрофона | — | — |
| `opening_door` (слой) | `lock.unlock` вызван, lock в переходе | Прогресс на контроле + overlay «Открываем…» | Отмена (если возможно) | повторный «Открыть» | по успеху → `door_opened` |
| `door_opened` (слой) | lock open / успех сервиса | overlay-успех «Открыто» `mdi:lock-open-check` (зелёный) | — | — | через 2–3с → возврат |
| `call_ended` | `call_state=ended` | «Вызов завершён» (кратко) | — | все | через 1–2с → скрытие/`idle` |
| `camera_unavailable` (слой) | стрим ошибка/EOF | Плейсхолдер `mdi:cctv-off` «Видео недоступно» | Повторить, прочие кнопки | — | авто-retry стрима (рефреш-на-открытии) |
| `audio_unavailable` (слой) | downlink 0 / audio fail | Видео + бейдж «Звук недоступен» | — | — | — |
| `microphone_permission_required` (слой) | `getUserMedia` denied / insecure origin | CTA «🎤 Разрешить микрофон» / «Нужен HTTPS» | кнопка разрешения | toggle микрофона | — |
| `error` | `call_state=error` / сбой сервиса | «Ошибка соединения» + причина | Повторить, Закрыть | — | — |

> **speaker_select из ТЗ — намеренно убран**: выбор аудиовыхода (earpiece) в вебе
> невозможен (§13). Остаётся `muted`-слой (Mute/Unmute). Это сознательное
> упрощение под реальность платформы, а не пропуск.

## 6. UI-структура: экран входящего вызова

```
┌─────────────────────────────────────────────┐
│ [🟢 домофон-иконка]  Подъезд 2          [✕]  │  ← шапка: имя домофона + закрыть
│ ● Входящий вызов · осталось 0:24             │  ← статус + обратный отсчёт окна
├─────────────────────────────────────────────┤
│                                               │
│            [  ВИДЕО ГОСТЯ (без звука)  ]       │  ← доминанта, 16:9, скруглённое
│                              [🔊 Включить звук]│  ← оверлей (появится после Принять)
│                                               │
├─────────────────────────────────────────────┤
│   (🔑)───────────►  Открыть дверь       🔓    │  ← Slide-to-open (наша карточка)
│            сдвиньте, чтобы открыть            │     или нативная кнопка/hold (DIY)
├─────────────────────────────────────────────┤
│        [  ✕ Отклонить  ]   [  ✓ Принять  ]   │  ← Отклонить (red) / Принять (green)
└─────────────────────────────────────────────┘
```

- **Иерархия сверху вниз:** имя+статус (контекст) → видео (главное) → Открыть
  (частое безопасное действие) → Принять/Отклонить (решение по вызову).
- **Открыть отделён** визуально (своя зона/разделитель) от Принять/Отклонить —
  чтобы не путать «открыть дверь» с «принять вызов».
- Принять — primary success (зелёный, справа = «позитив/вперёд»); Отклонить —
  error (красный, слева). Открыть — **нейтрально-акцентный, НЕ красный**.

## 7. UI-структура: экран активного разговора

```
┌─────────────────────────────────────────────┐
│ [🟢]  Подъезд 2                         [✕]  │
│ ● Разговор активен              ⏱ 00:32      │  ← статус «активен» + таймер
├─────────────────────────────────────────────┤
│            [  ВИДЕО + ЗВУК ГОСТЯ  ]           │  ← тап по видео = unmute (если muted)
├─────────────────────────────────────────────┤
│   (🔑)───────────►  Открыть дверь       🔓    │
│            сдвиньте, чтобы открыть            │
├─────────────────────────────────────────────┤
│   [🎤 Микрофон вкл]  [🔊 Звук]  [☎ Завершить] │  ← mic toggle / mute / завершить(red)
└─────────────────────────────────────────────┘
```

- **Отличие от входящего** (требование ТЗ): синий/нейтральный заголовок «Разговор
  активен» вместо красного «Входящий», **таймер** вместо отсчёта окна, **нет
  «Принять»**, появляются Микрофон + Звук, фон-акцент чуть другой (active-state).
- **Состояние микрофона явно видно**: вкл — заполненная иконка `mdi:microphone` +
  «вкл»; выкл — `mdi:microphone-off` + приглушённый цвет + «выкл».
- **Звук**: если muted (политика автоплея) — кнопка/оверлей «Включить звук»
  заметны; если играет — обычный mute-toggle.

## 8. Логика кнопок

| Кнопка | Где | Действие (реальное) | Цвет/роль | Условие/блок |
|---|---|---|---|---|
| **Принять** | incoming | `elektronny_gorod.answer` | success (green), primary | только `ringing`; исчезает после ответа |
| **Отклонить** | incoming | `elektronny_gorod.hangup` (decline/CANCEL) | error (red) | только `ringing`/`connecting` |
| **Завершить** | active | `elektronny_gorod.hangup` (BYE) | error (red) | только `active`/`connecting` |
| **Открыть** | incoming+active | `lock.unlock` (lock домофона) | нейтрально-акцентный | всегда, пока есть домофон; **независимо от SIP** |
| **Микрофон** | active | card-local `getUserMedia` → WS `elektronny_gorod/intercom_uplink` | toggle | secure origin + permission (§13) |
| **Звук (Mute)** | active | `<video>.muted` toggle / unmute-жест | toggle | — |
| **✕ Закрыть** | all | свернуть карточку/закрыть popup (НЕ hangup) | tertiary | не завершает вызов |

🔑 **Reject и End — один сервис `hangup`** (контекст определяет CANCEL vs BYE). Это
не баг — так же мост ведёт себя в `call_controller.py`.

🔑 **«Открыть» ≠ «Принять»:** разные взаимодействия (slide/hold vs tap), разные
зоны, разные цвета, разные иконки (🔑 vs ✓). Открыть не зависит от состояния SIP.

## 9. UX безопасности: кнопка «Открыть»

> Владелец: «не вижу большой проблемы в случайном открытии» → защита нужна
> умеренная, не параноидальная. Делаем **два варианта** (DIY-нативный + наш slide).

### Варианты (как просили — 2–3)

| Вариант | Реализация в HA | Защита | Облик | Вердикт |
|---|---|---|---|---|
| **1. Кнопка + подтверждение** | `tap_action` + `confirmation:` (стоковый tile/button, **zero-code**) | Нативный диалог «Открыть дверь?» | 100% родной HA | ✅ для DIY/стока |
| **2. Long-press (hold)** | `hold_action: lock.unlock` (стоковый, **zero-code**) | Удержание ~600мс + прогресс | Родной HA | ✅ для DIY/стока |
| **3. Slide-to-open** | Кастом (Lit, на стиле `ha-control-slider`) | Свайп + порог + прогресс | Привычно по приложению | ✅ в нашей карточке |

### Выбор

- **DIY/toolkit-путь:** рекомендуем **long-press (hold)** как дефолт — нативно,
  zero-code, есть прогресс, трудно нажать случайно; **tap+confirmation** как
  альтернатива для тех, кто хочет максимально явный диалог.
- **Наша карточка (`eg-intercom-call-card`) — адаптивно по типу ввода:**
  - 🔴 **Slide неудобен на десктопе с мышью** (по замечанию владельца). Поэтому:
    **на тач-устройствах** (телефон/планшет/настенная панель, `pointer: coarse`) —
    дефолт **Slide-to-open**; **на десктопе** (`pointer: fine`/мышь) — дефолт
    **hold или tap+confirmation** (свайп мышью не навязываем).
  - Конфиг `open_action: auto | slide | hold | tap` (дефолт `auto` = адаптив выше;
    любой можно зафиксировать явно).

**Почему slide хорош на тач (но не на десктопе):** (1) знаком по приложению;
(2) один жест без отдельного диалога; (3) **визуальный прогресс** открытия;
(4) удобен одной рукой на тач-панели; (5) на стиле `ha-control-slider` — не выглядит
инородно. На десктопе же свайп мышью неудобен → там hold/tap.

### Процесс / успех / ошибка открытия

- **Процесс (`opening_door`):** контрол показывает прогресс/«Открываем…», блок
  повторного нажатия; (опц.) короткая задержка для отмены.
- **Успех (`door_opened`):** контрол → зелёный «Открыто» + overlay `mdi:lock-open-check`,
  держим 2–3с, затем возврат. (Зеркалит «Открыто» приложения.)
- **Ошибка (`lock.unlock` fail/timeout):** контрол → красный, overlay `mdi:lock-alert`
  «Не удалось открыть» + «Повторить». Никогда не показываем «успех» без подтверждения
  от `lock`.

## 10. Адаптивность

Карточка респонсивна по **ширине контейнера** (`ResizeObserver`/container-queries),
а не по вьюпорту — корректно и в узкой колонке дашборда, и в fullscreen.

### Мобильный телефон (≤480px)
- Вертикальный layout; видео — 16:9 на всю ширину, главный элемент.
- Кнопки крупные (≥48dp), основные действия внизу (зона большого пальца).
- Управление одной рукой; Открыть — над рядом Принять/Отклонить.
- Безопасные зоны (notch/gesture bar) — отступы.

### Планшет (≤1024px)
- Видео крупное; на ландшафте — видео слева, кнопки справа (две колонки).
- Тач-цели и текст читаемы с расстояния; больше воздуха.

### Настенная панель (always-on, fullscreen)
- **Очень простой** экран: видео максимально, 2–3 крупные кнопки, минимум мелочей.
- Имя домофона и статус — крупно сверху.
- **Fullscreen при звонке:** авто-навигация на выделенный view (zero-dep) ИЛИ
  `browser_mod.popup` (поверх всего). Авто-unmute на kiosk-браузере возможен, если
  пользователь уже взаимодействовал/включён autoplay-flag (§13).

### Десктоп (≥1024px)
- Компактная карточка на дашборде в `idle`; при звонке — **модалка**
  (`browser_mod`, опционально) или яркая подсветка карточки.
- Все действия без прокрутки; hover-состояния (но не полагаемся на hover).
- 🔴 **Открытие двери — hold или tap+confirmation, НЕ slide** (свайп мышью неудобен,
  по замечанию владельца).

### Компактный режим Lovelace
```
┌───────────────────────────────┐
│ [мини-видео]  Подъезд 2        │
│  ● Входящий      ⏱0:24         │
│ [Открыть] [Отклон.] [Принять]  │
└───────────────────────────────┘
```
- Только важное: имя, мини-превью, статус, основные кнопки. `getCardSize()` мал.

### Полноэкранный режим
- Видео занимает максимум; кнопки — полупрозрачной панелью снизу поверх видео или
  под ним. **Элементы не перекрывают лицо/центр кадра** (панель — нижние ~20%, со
  скримом-градиентом для читаемости).

## 11. Визуальная иерархия

1. **Главный элемент — видео гостя** (всегда, и до приёма). Центр/верх, максимум площади.
2. **Имя домофона + статус** — сверху, контекст «кто и откуда».
3. **Открыть дверь** — отдельная зона между видео и кнопками вызова (частое
   безопасное действие, отделено разделителем).
4. **Принять/Отклонить** (incoming) или **Микрофон/Звук/Завершить** (active) — внизу.
5. Дополнительное (Закрыть, таймер) — мелко, в углах.

**Разделение основных и второстепенных:** primary-действие на экране — одно
(Принять на incoming; Завершить на active). Остальные — визуально подчинены
(меньше/тоньше/нейтральнее). Открыть — заметный, но через **иную форму** (слайдер),
не конкурирует цветом с primary.

**Цвета действий:** Принять = success/green; Отклонить+Завершить = error/red;
Открыть = нейтрально-акцентный (НЕ red); Микрофон/Звук = state-on/off акценты.

## 12. Рекомендации для светлой и тёмной темы

**Только theme-переменные, никаких хардкод-hex** (иначе инородно + ломается в темах):

| Элемент | Токен |
|---|---|
| Фон карточки | `--ha-card-background`, `--card-background-color` |
| Текст заголовка/боди | `--primary-text-color` / `--secondary-text-color` |
| Принять | `--success-color` (fallback `var(--label-badge-green)`) |
| Отклонить/Завершить | `--error-color` |
| Открыть (акцент) | `--primary-color`; завершённое «Открыто» → `--success-color` |
| Слайдер: трек | `rgba(var(--rgb-primary-text-color), 0.1)` |
| Слайдер: ручка | `--primary-color` → `--success-color` по завершении |
| Статус «Входящий» | `--error-color`/`--warning-color` (внимание) |
| Статус «Разговор активен» | `--info-color`/`--primary-color` |
| Скрим оверлеев | `rgba(0,0,0,0.55)` (читаемо в обеих темах) |
| Иконки | `currentColor` / `--state-icon-color` |

- Видео-леттербокс — чёрный в обеих темах (нормально для видео).
- Контраст текста на оверлеях/видео — ≥4.5:1 (скрим-градиент обязателен поверх кадра).
- Светлую и тёмную проверять **отдельно** (бордеры/disabled не должны исчезать).
- **Не тащить салатовый брендинг ЭГ** — наследуем тему пользователя (родной облик).

## 13. Технические ограничения (HA + браузеры)

### Видео / WebRTC
- **Решение: видео встраиваем в карточку нативно** — через HA-native WebRTC
  (встроенный в HA go2rtc, WS-сигналинг `camera/webrtc/offer`), **без обязательной
  зависимости** от `webrtc-camera`. Источник — `camera.intercom_call`
  (video+downlink-audio). go2rtc — в LAN, креды у HA, фронт за HA-логином →
  **4G работает без экспозиции**.
- `webrtc-camera` — **необязательный fallback** (если у пользователя уже стоит и
  HA-native по какой-то причине не подойдёт); как зависимость не требуется
  (функционально ограничена — по решению владельца).
- **EOF** (протухший operator-URL) снят **рефрешем-на-открытии** `stream_source()`.
- `picture-entity`/инлайн-MSE **глушит звук** → для звука нужен WebRTC-плеер
  (наша карточка использует HA WebRTC-offer напрямую или встраивает webrtc-camera).

### Автозвук (главное ограничение UX)
- Политика автоплея (Chrome/Safari/WebView) **блокирует автозапуск со звуком** →
  `<video>` стартует `muted`. `muted:false` не помогает до **жеста пользователя**.
- Решение: **оверлей «🔊 Включить звук»**; тап по «Принять»/видео = жест (браузер
  помнит на сессию). На kiosk-браузере панели можно включить autoplay-flag.
- **Авто-звук по умолчанию гарантировать нельзя** — честно отражаем в UX.

### Микрофон (uplink)
- `getUserMedia` требует **secure context (HTTPS)** + **разрешение**. На `http://`
  (часто локальный HA) микрофон **недоступен** → нужен HTTPS-доступ.
- **Решение (по владельцу): при переходе в `active` микрофон захватывается
  автоматически**, чтобы не делать лишний тап — но **только если** разрешение уже
  выдано (`navigator.permissions.query({name:'microphone'})==='granted'`) **и**
  `isSecureContext`. Если разрешение ещё не выдано — показываем CTA «Разрешить
  микрофон» (один доп. тап, это приемлемо). Микрофон **всегда можно выключить**
  тоглом во время разговора.
- Тап «Принять» — сам по себе user-жест → заодно снимает блок автоплея звука гостя
  (один жест включает и слух, и голос).
- Транспорт микрофона: браузер → AudioWorklet (Int16 PCM) → **бинарный фрейм по
  авторизованному HA-WS** → WS-команда `elektronny_gorod/intercom_uplink` → UplinkSink
  → SIP RTP (ADR-0013, механизм #1; **уже реализован** в `uplink_ws.py` +
  `eg-intercom-mic-card.js`). Это **отдельный канал** от downlink-WebRTC (не go2rtc
  backchannel — тот Slice 2/future).
- **Эхо/дуплекс:** downlink (динамик) и uplink (микрофон) на одном устройстве →
  риск эха. Включаем `echoCancellation/noiseSuppression/autoGainControl` (уже в
  карточке); опция half-duplex (push-to-talk) для проблемных кейсов.

### Аудиовыход
- Маршрутизация в **earpiece/«телефон» недоступна** веб-странице. `setSinkId()`
  (выбор устройства вывода) ограничен (Chrome-only, не в Companion WebView) → **не
  закладываем**. Остаётся Mute/Unmute + системная громкость.

### Тайминги / жизненный цикл
- **Окно дозвона ~30с** (`CallInvalidated`, фаза `ringing`): карточка авто-истекает;
  «Принять» гаснет после окна; домофон сбрасывает сам.
- **Лимит разговора ~1 мин** (фаза `active`): звонок завершается оператором/панелью
  примерно через минуту → карточка должна штатно обработать `ended`; (опц.) тонкий
  индикатор в последние ~10с.
- **Таймер на экране — открытый вопрос (TBD live).** В приложении это **секундомер
  (счёт вверх)**, не обратный отсчёт. Делаем его **конфигурируемым и по умолчанию
  ненавязчивым** (или скрытым), решение — после live-проверки «не раздражает ли».
- Ответ строго привязан к `Call-ID` из FCM (сервер не отвечает на устаревший вызов).
- `REGISTER→INVITE→200 OK` мгновенный (<1с) → `connecting` короткий.

### Фронтенд / Lovelace
- **Нет стокового slide-to-confirm** и **нет stock fullscreen-takeover из карточки**
  → slide = кастом; fullscreen = авто-навигация (zero-dep) или `browser_mod`.
- Несколько вкладок/браузеров: `call_state` общий (сервер), а **микрофон/mute —
  свои в каждом браузере** (это корректно, не баг).
- Кастомные карточки регистрируются как Lovelace-ресурс
  (`/elektronny_gorod_static/eg-intercom-call-card.js`).

## 14. Home Assistant entities / services — архитектура

Принцип: сущности — **полный toolkit** (DIY-путь) и **бэкенд нашей карточки**
одновременно. Микрофон/mute — **намеренно client-side** (привязаны к браузеру, не к
серверу — иначе «чей микрофон?» неоднозначно).

### Обязательные (есть / добавить)

| Сущность/сервис | Статус | Назначение |
|---|---|---|
| `camera.<intercom>` | есть | Видео домофона (peacetime, без звука) — для `ringing` |
| `camera.intercom_call` | есть (Slice 1) | Видео **+ звук гостя** на время активного вызова; вне вызова unavailable |
| `event.<intercom>_doorbell` | есть | `ring`/`ended` (device_class doorbell) — триггер показа |
| `lock.<intercom>` | есть | Открытие двери (`lock.unlock` = accessControlOpen) |
| `sensor.<intercom>_call_state` | **NEW (рекоменд.)** | enum `idle/ringing/connecting/active/ended/error` + атрибуты `call_id, intercom_name, camera_entity, lock_entity, started_at` — **единый источник истины** для карточки и автоматизаций |
| `elektronny_gorod.answer` / `.hangup` | есть | Принять / Отклонить-Завершить |
| WS `elektronny_gorod/intercom_uplink` | есть (Slice 2) | Микрофон (uplink), client-side |

### Опциональные (расширенный UX / DIY-эргономика)

| Сущность | Зачем | Примечание |
|---|---|---|
| `button.<intercom>_answer` / `_hangup` | ~~entity-обёртки сервисов~~ | **Решено: не делаем** — `button`-карточка, вызывающая сервис `answer`/`hangup`, и так работает (стандарт HA). Добавим по запросу DIY-юзеров |
| `sensor.<intercom>_call_duration` | кому нужна длительность сущностью | иначе карточка считает от `started_at` (не спамит state) |
| `switch.*` DND | есть | «не беспокоить» (не часть экрана вызова) |

### НЕ-сущности (по дизайну — client-side)

- **Микрофон вкл/выкл** — состояние браузера (`getUserMedia`), не сервер-сущность.
- **Mute/Unmute** — свойство `<video>` (политика автоплея), client-side.
- **Аудиовыход** — невозможен в вебе (§13), не моделируем.

### Что сервисами вместо сущностей

Принять/Отклонить/Завершить — **сервисы** (`answer`/`hangup`), т.к. они контекстны
(привязка к активному `Call-ID`). Button-обёртки — опциональны для удобства DIY.

> **Реконсиляция с текущим дашбордом:** сейчас используется `input_boolean ... state:
> on` + conditional-cards. Рекомендация — заменить helper на `sensor.*_call_state`
> (чище, без ручных automation-toggle). Это backend-добавление сущности (план
> отдельным slice'ом).

## 16. Acceptance criteria

- [ ] Видео звонящего видно **до** принятия вызова (`ringing` → камера домофона).
- [ ] Видно имя домофона и статус (`Входящий` / `Разговор активен`).
- [ ] Можно **Принять** (`answer`), **Отклонить** (`hangup`).
- [ ] Можно **открыть дверь без принятия** вызова (`lock.unlock`, независимо от SIP).
- [ ] Можно открыть дверь **во время разговора**.
- [ ] После принятия кнопка **«Принять» исчезает**, появляются Микрофон/Звук/Завершить.
- [ ] Экран разговора **визуально отличается** от входящего (статус, таймер, набор
      кнопок, акцент).
- [ ] Микрофон **вкл по умолчанию, только если разрешение уже выдано и origin secure**;
      иначе — понятный CTA «Разрешить микрофон».
- [ ] Если микрофон недоступен — понятное сообщение, кнопка disabled (без «битого» вида).
- [ ] Если звук не запускается авто (политика браузера) — **явная кнопка «Включить звук»**.
- [ ] Кнопка открытия имеет **защиту от случайного нажатия**, адаптивную по вводу:
      тач → slide, десктоп/мышь → hold/tap+confirm (slide на десктопе не используется).
- [ ] Микрофон при `active` **захватывается автоматически** (если permission granted +
      secure) и **может быть выключен** тоглом; иначе — CTA «Разрешить микрофон».
- [ ] Состояние открытия (процесс) **отображается**; успех — только по подтверждению `lock`.
- [ ] Ошибка открытия **отображается** с «Повторить».
- [ ] Окно 30с: «Принять» **гаснет** по истечении; карточка авто-скрывается.
- [ ] Корректно работает на **мобильном** и **десктопе**; адаптив по ширине контейнера.
- [ ] Поддержка **светлой и тёмной** темы HA (только theme-токены, проверено в обеих).
- [ ] Элементы управления **не перекрывают** лицо/центр кадра (скрим-панель снизу).
- [ ] Понятно **без инструкции** (родной HA-облик, привычные паттерны).
- [ ] **Toolkit-путь:** все сущности (`call_state`, camera, lock, event, services)
      позволяют собрать рабочий экран на стоковых карточках без нашей карточки.
- [ ] **Из-коробки:** `eg-intercom-call-card` подключается как один Lovelace-ресурс
      и работает на 4G (видео+звук инлайн, без экспозиции go2rtc).

## 17. Открытые вопросы — решения владельца (2026-06-24)

**Решено:**
1. ✅ **`sensor.*_call_state`** — **добавляем** (отдельный slice). Единый источник
   состояния для карточки + автоматизаций; заменяет `input_boolean`-helper.
2. ✅ **`button.*_answer/_hangup`** — **не делаем**. Сервисов `answer`/`hangup`
   достаточно (button-карточка вызывает сервис — стандарт HA). Добавим по запросу.
3. ✅ **Видео в карточке** — **HA-native WebRTC напрямую**, без обязательной
   `webrtc-camera` (она функционально ограничена). `webrtc-camera` — опц. fallback.
4. ✅ **Fullscreen на панели** — **опциональная** фича (не из коробки для всех).
   Базово — авто-навигация на view; `browser_mod.popup` — если получится, как опция.
5. ✅ **Микрофон** — **авто-захват при `active`**, если разрешение уже выдано и origin
   secure (без лишнего тапа). Если нужно выдать разрешение — один доп. тап OK.
   Микрофон **всегда выключаемый** тоглом. (PTT — не дефолт.)
6. ✅ **Таймер** — TBD live: в приложении это **секундомер (счёт вверх)**, не обратный
   отсчёт; делаем конфигурируемым, по умолчанию ненавязчивым/скрытым. Окно дозвона
   30с — авто-скрытие; обратный отсчёт показывать по необходимости.
7. ✅ **call_duration** — не отдельная сущность; карточка считает от `started_at`
   (опц. сенсор позже, если попросят).
8. ✅ **Несколько активных вызовов** — **«один за раз»** (как приложение); очередь/
   мультивызов — **в бэклог развития**, не сейчас.

**Дополнительное замечание владельца (учтено в §9/§10):** slide-to-open **неудобен в
веб-HA на десктопе** (мышь) → `open_action: auto` (тач→slide, десктоп→hold/tap).

**Реально открытых вопросов к разработчику не осталось** — можно переходить к плану
реализации (backend `sensor.*_call_state` + Lit-карточка).

## Приложение B. Фронтенд-стек карточки

- **Lit + TypeScript** (язык фронтенда самого HA → нативный облик, читаемость,
  поддержка), **не чистый JS** (по решению владельца). Текущая
  `eg-intercom-mic-card.js` (ванильный JS) → **поглощается** новой Lit-карточкой
  (логика микрофона переносится как есть: AudioWorklet + WS-uplink).
- **Компоненты HA:** переиспользуем `ha-card`, `ha-control-button(-group)`,
  `ha-control-slider`, `ha-icon`, `ha-camera-stream` / WebRTC-offer — чтобы карточка
  была неотличима от родных.
- **Сборка:** Rollup/Vite → один бандл `eg-intercom-call-card.js`; раздаётся как
  статик-ресурс интеграции (как mic-карточка сейчас), регистрируется Lovelace-ресурсом.
- **Конфиг карточки (черновик):**
  ```yaml
  type: custom:eg-intercom-call-card
  call_state: sensor.podyezd_2_call_state   # единый источник (или auto-discovery)
  camera: camera.intercom_call               # видео+звук активного вызова
  doorbell_camera: camera.podyezd_2          # видео в ringing
  lock: lock.podyezd_2                        # открытие двери
  open_action: auto                          # auto|slide|hold|tap (auto: тач→slide, мышь→hold/tap)
  mic: true                                   # показывать микрофон
  mic_autostart: true                         # авто-захват при active, если permission granted
  autoplay_audio_hint: true                   # оверлей «Включить звук»
  timer: auto                                 # auto|stopwatch|off (TBD live)
  ```
- **Разделяемая логика** (call-state-машина, таймер, mic, open-control) — в модулях,
  чтобы DIY-путь мог переиспользовать отдельные web-components при желании.
