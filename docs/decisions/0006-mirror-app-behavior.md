# ADR-0006: Mirror application behavior

- **Status:** accepted
- **Date:** 2026-05-23
- **Owner:** @gentslava + Lead Architect Agent

## Context

`myhome.proptech.ru` — закрытое API мобильного приложения «Мой Дом» (Электронный город) / «Умный Дом.ру». Оператор **не предоставляет** third-party API, нет публичной документации. Интеграция работает в режиме эмуляции оригинального мобильного клиента: те же endpoints, тот же User-Agent, тот же auth flow.

В ходе работы над AIDD-документацией возник соблазн фиксировать «гипотезы про неизвестные эндпоинты» (refresh-token endpoint, webhook, push). Это плохо для проекта потому что:

- Запросы вне поведения реального приложения = аномалия для backend оператора.
- Спекуляции в docs провоцируют экспериментировать («попробую GET /events») — это засветится в трафике.
- В случае массового несоответствия трафика — риск блокировки доступа / капчи / отзыва токена / юридических претензий.

Owner озвучил это как фундаментальный constraint: «суть интеграции в эмуляции работы приложения. Если начнутся аномальные запросы, не соответствующие логике работы приложения, оператор может обратить внимание и ограничить доступ».

## Decision

**Любое поведение интеграции должно зеркалировать оригинальное мобильное приложение.**

### Что это означает на практике

1. **Endpoints, методы, headers, частота** — только те, что наблюдались в HAR-снимках реального приложения.
2. **Новые фичи** в интеграции добавляются **только** если соответствующий сценарий есть в приложении и собран HAR.
3. **Никаких «гипотез»** в архитектурной документации. Если что-то неизвестно — «требует подтверждения через HAR», а не «предположим, что ...».
4. **User-Agent эмуляция** ([`user_agent.py`](../../custom_components/elektronny_gorod/user_agent.py)) — обязательная часть каждого запроса, формат строго соответствует приложению.
5. **Частота polling** в coordinator (когда [ADR-0002](0002-coordinator-pattern.md) будет реализован) должна быть **не выше**, чем у приложения в фоне.
6. **refresh_token**: статус — **unknown**. В HAR-сессиях, наблюдавшихся owner-ом, использование не зафиксировано. До подтверждения через HAR со сценарием истечения access_token — не реализовывать auto-refresh.

### Что это исключает

- Использование endpoints, не наблюдавшихся в HAR.
- Параллельные запросы, если приложение делает их последовательно (и наоборот).
- «Optimization» через дедупликацию запросов, если приложение делает их каждый раз.
- Свои headers / query-параметры, которых нет в приложении.

### Workflow для новых фич

1. **HAR-collection** — записать сессию приложения, выполняющую целевой сценарий (см. [`runbooks/har-collection.md`](../aidd/runbooks/har-collection.md)).
2. **Analyse** — выписать endpoints, методы, headers, тайминги, последовательности.
3. **Spec** — `docs/features/<id>/prd.md` с привязкой к HAR-snapshot.
4. **Implementation** — slice by slice, повторяя поведение приложения.
5. **Verify** — сравнить трафик от нашей интеграции с HAR приложения (опционально, через тот же Charles).

## Consequences

### Positive

- Снижает риск блокировки доступа от оператора.
- Делает интеграцию **предсказуемой** — поведение совпадает с тем, что owner может проверить в приложении.
- Чёткая граница для AI-агентов: «если нет в HAR — не добавлять».
- Реверс-инжиниринг становится явной частью процесса разработки, не неявной интуицией.

### Negative

- Каждая новая фича = HAR-сессия + анализ. Это удлиняет cycle time.
- Сложно реализовать функциональность, которой нет в приложении (например, агрегацию по местам).
- Зависимость от наличия физического устройства / эмулятора для HAR-collection.
- HAR содержит секреты — нужна процедура redaction для шаринга.

### Mitigation

- Накапливать HAR-снимки в [`research/api/`](../../research/api/) (локально, не в репо).
- Runbook [`har-collection.md`](../aidd/runbooks/har-collection.md) описывает процедуру (APK rebuild с MITM-патчем, Charles, redaction).
- При невозможности собрать HAR — фича откладывается, не реализуется наспех.

## Alternatives considered

1. **Свободный реверс + эксперименты с endpoints.** Отклонено — основная причина этого ADR.
2. **Зеркалирование только critical path (auth + read), эксперимент в остальном.** Отклонено — backend оператора может коррелировать все запросы одного клиента; даже один аномальный запрос привлекает внимание.
3. **Полностью отказаться от reverse engineering, использовать official API.** Отклонено — official API не существует.
