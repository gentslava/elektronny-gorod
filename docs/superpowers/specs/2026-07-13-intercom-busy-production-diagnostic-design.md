# Production-диагностика захвата второго домофона

**Дата:** 2026-07-13  
**Статус:** согласованный дизайн, до реализации  
**Область:** только диагностика SIP-вызова; открытие двери и ответ на вызов исключены

## Цель

Установить, какое действие интеграции заставляет физический домофон B самостоятельно
показывать «Занято» и блокировать управление, пока идёт вызов с домофона A.

Нужно сравнить три режима при неизменном окружении и работающих штатных приложениях:

1. текущая интеграция с полным `mint → REGISTER → INVITE → 100 Trying`;
2. интеграция с FCM/HA-событиями, но без SIP-действий;
3. полностью выгруженная config entry, то есть только штатные приложения.

После первого реального вызова дополнительно проверяются изолированные `mint` и два
варианта `REGISTER` вне активного звонка.

## Контроль переменных

- Штатные приложения на всех телефонах остаются установленными и работающими во всех
  режимах.
- Все три вызова выполняются с одного домофона A в одну квартиру; наблюдается один и
  тот же домофон B.
- Различается только поведение нашей Home Assistant integration.
- SIP-трафик интеграции отделяется по адресу и локальному UDP-порту Home Assistant.
- Первый FCM `ring` фиксирует `place_id`, `access_control_id` и `call_id`; эти значения
  используются только внутри diagnostic runner и не печатаются вместе с секретами.

## Запуск пользователем

1. Пользователь пишет `пошёл`.
2. Runner переходит в `armed` на 20 минут, не меняя текущего поведения интеграции.
3. Пользователь без спешки приходит к домофону A.
4. Первый следующий FCM `ring` запускает эксперимент и становится временем `T0`.
5. Пользователь запускает телефонный таймер одновременно с нажатием вызова; задержка
   между физическим нажатием и `T0` учитывается по production-логу.
6. Если `ring` не пришёл за 20 минут, runner снимается без изменений.

## Временная шкала

| Время от T0 | Действие | Что делает пользователь |
|---:|---|---|
| `0:00` | Режим 1: текущая полная интеграция | Идёт от A к B, проверяет «Занято» и блокировку управления |
| `~0:30` | Реальный вызов завершается штатно | Остаётся у B |
| `0:50` | Только API `mint_sip_device`; без `REGISTER` | Смотрит, изменился ли B |
| `1:20` | Текущий `REGISTER`; затем явный unregister | Смотрит, появилось ли «Занято» повторно |
| `1:55` | REGISTER формата штатного приложения; затем unregister | Смотрит, появилось ли «Занято» повторно |
| `2:20` | Runner включает режим 2: FCM/HA без SIP | Возвращается к A |
| `3:00` | Вызов №2 | Снова идёт к B и проверяет поведение |
| `4:30` | Config entry выгружается; начинается режим 3 | Возвращается к A |
| `6:00` | Вызов №3, только штатные приложения | Идёт к B и проверяет поведение |
| `7:30–8:00` | Config entry загружается обратно | Ничего не делает |

Тайминг переключается от фактического `T0`, а не от момента ухода пользователя от
компьютера. Отклонение до нескольких секунд допустимо и будет видно в журнале.

## Изолированные пробы после вызова №1

### Mint only

Выполняется только POST `sipdevices` для того же `place_id/access_control_id` и той же
installation identity. SIP-сокет не создаётся.

### Current REGISTER

Используется текущий builder интеграции: `app-id`, `pn-type`, `pn-tok`, но без
проприетарного Contact-параметра `Call-Id:%20{FCM_CALL_ID}`. Первичный `Expires: 30`
снимается явным REGISTER с `Expires: 0` после короткого окна наблюдения.

### Official-format REGISTER

Используется тот же SIP login/realm, но весь REGISTER зеркалит сохранённый эталон
штатного приложения: Contact включает `Call-Id:%20{FCM_CALL_ID}`, а заголовки включают
эталонные `Supported` и `Accept: application/sdp`. Первичный `Expires: 30` также
снимается явным `Expires: 0`.

Обе REGISTER-пробы не отвечают `100`, `180` или `200 OK` на INVITE. Если после уже
завершённого вызова неожиданно приходит INVITE, runner отвечает финальным безопасным
`480 Temporarily Unavailable`, снимает регистрацию и помечает пробу неоднозначной.

## Режимы второго и третьего вызова

### Режим 2 — FCM/HA без SIP

Интеграция продолжает принимать и разбирать FCM, создавать doorbell/call-state события
и соблюдать штатные таймеры. Для `ring` полностью запрещены `mint_sip_device`,
`REGISTER`, ожидание INVITE и `100 Trying`.

### Режим 3 — official app only

Config entry выгружается через Home Assistant API до вызова №3. FCM listener,
координатор и SIP-controller интеграции отсутствуют. Отдельный watchdog, не связанный
с lifecycle entry, загружает entry обратно после окна теста.

## Наблюдаемость и секреты

- HA-журнал и SIP-capture начинаются до `armed` и получают монотонные stage markers.
- В итог выводятся только время, направление, SIP method/status, panel/access-control
  correlation и длительность этапа.
- Access/refresh tokens, Digest Authorization, SIP password, FCM token и полные headers
  не выводятся.
- Если для точного сравнения нужен raw pcap, он хранится только во временном root-only
  файле на production host и удаляется после получения санитизированной сводки.

## Безопасность и rollback

- Runner никогда не посылает `200 OK` на INVITE, RTP, DTMF или команду открытия двери.
- Каждая диагностическая регистрация завершается `Expires: 0` и закрытием UDP-сокета.
- Любая ошибка этапа отменяет следующие активные пробы и переводит систему в rollback.
- Независимый watchdog восстанавливает исходный код/режим и загружает config entry не
  позднее `T0 + 9 минут`.
- Если runner завершается до `T0`, production state не меняется.
- Перед приглашением пользователя к панелям runner проходит локальные тесты и dry-run
  без сетевых POST/REGISTER.

## Интерпретация результата

| Наблюдение | Вывод |
|---|---|
| «Занято» только в режиме 1 | Причина находится в live SIP-цепочке интеграции |
| «Занято» появляется на `mint only` | Побочный эффект создаёт API `sipdevices` |
| «Занято» появляется только на current REGISTER | Вероятна ошибка scope/Contact текущей регистрации |
| Оба REGISTER вызывают «Занято» | Сам факт регистрации влияет на panel session; `Call-Id` недостаточен |
| Ни одна post-call проба не вызывает «Занято» | Нужен живой INVITE и/или `100 Trying`; потребуется отдельный staged-live вызов |
| «Занято» остаётся в режиме 2 | Причина не в `mint/register_and_hold`; сравниваем с режимом 3 |
| «Занято» есть в режиме 3 | Поведение воспроизводится без нашей integration и требует пересмотра baseline |

## Не входит в этот эксперимент

- Исправление production-кода до подтверждения причины.
- Автоматический ответ, аудио/RTP и открытие двери.
- Доказательство роли `100 Trying` вне живого INVITE: старый SIP transaction нельзя
  корректно воспроизвести после завершения звонка. При необходимости это будет отдельный
  эксперимент с дополнительным реальным вызовом.
