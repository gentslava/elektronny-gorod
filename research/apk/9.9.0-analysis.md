## What changed since the baselines

### Common changes in both flavours

- New Stories/banner communication layer, including advertising labels,
  advertiser metadata and metrics.
- New VPN/service-pipe warning screen. The client recognizes an HTML block
  page by `REQUEST-ID`, IP and datetime fields and shows a dedicated warning
  that the app is unavailable outside Russia or while VPN is enabled.
- Access-key notification status DTO and settings strings.
- No permission changes. Target SDK stays 36.
- New activities: `VpnWarningActivity` and `StoriesActivity`.

The 9.9.0 first-party class inventories converge: 952 shared paths; the only
flavour-only path is Smart Home's generated `R.java`. The relevant static
endpoint-string sets are identical (449 each), so both apps now use one very
close shared codebase with branding/auth differences.

### Мой Дом 9.7.0 → 9.9.0

- `versionCode`: `90700000` → `90900020`.
- `minSdk`: 29 → 32 (Android 12L/API 32).
- 31 first-party class paths added, none removed.
- New access-key notification preferences and Stories/VPN UI.

### Умный Дом.ру 9.3.0 → 9.9.0

- `versionCode`: `90300010` → `90900020`.

## Runtime API findings — Мой Дом 9.9.0

### Auth flows

Password flow:

1. `GET /auth/v2/login/{login}` → 200.
2. `POST /auth/v2/auth/{login}/password` with
   `{login,timestamp,hash1,hash2}` → token response.

SMS flow:

1. `GET /auth/v2/login/{phone}` → 300 and contract list.
2. `POST /auth/v2/confirmation/{phone}` with contract/place fields → 200.
3. `POST /auth/v3/auth/{phone}/confirmation` with
   `{accountId,operatorId,subscriberId,login,confirm1,confirm2}` → token response.

These contracts match `api.py`; no auth migration is required.

### Bootstrap and main screens

Observed after auth:

- `GET /rest/v3/subscriber-places`.
- `GET /rest/v1/places/{placeId}/cameras`.
- `GET /rest/v1/places/{placeId}/accesscontrols`.
- `GET /rest/v2/places/{placeId}/public/cameras`.
- `GET /rest/v1/places/{placeId}/screen-sections`.
- `GET /api/mh-customer/mobile/v1/customers/places/{placeId}/settings/screens`.
- `GET /rest/v1/stomp/available-features` and WebSocket `/events` (101).
- one `POST .../sipdevices` per access control.
- Events tab: `POST /rest/v1/events/search?page&sort` with `{placeIds}`.
- Cabinet/balance: `GET /api/mh-payment/mobile/v1/finance?placeId`.

The integration already uses every endpoint needed for its current entity
model. Events history remains a documented optional feature gap.

### New/changed exact contracts

- New communications bootstrap:
  `POST /api/mh-customer-communication/mobile/v1/communications` with
  `{appId,appVersion,appVersionCode,platform}` → `{banners,stories}`.
  It is app UI content and has no current HA entity use-case.
- Live video now explicitly sends
  `GET /rest/v1/forpost/cameras/{id}/video?LightStream=0&Format=H264`.
- `device-installations` is pre-auth, carries no Bearer and omits
  `deviceType`; `subscriberNotifications` is post-auth and includes
  `deviceType: MOBILE_APPLICATION`.

### PCAP-only transport findings

The provided 147.6-second PCAP is encrypted for HTTP but confirms:

- `myhome.proptech.ru` over TLS/443;
- three TLS video connections to `forpost-15.novotelecom.ru:18081`
  (about 7.5 MB inbound total);
- three parallel SIP registrations over UDP/5060, followed by Digest 401,
  successful 200 registration, NOTIFY traffic and renewals about every 27 s;
- server OPTIONS receive 400 responses from the stock client in this capture.

This persistent multi-registration differs intentionally from the
integration's register-on-ring strategy (ADR-0012); the current HA design
avoids permanent SIP bindings and relies on FCM for the ring trigger.

## Integration impact

Applied:

- runtime app identity → `9.9.0` / `90900020`;
- live video query mirrors explicit `Format=H264`;
- public device installation omits Bearer and `deviceType`;
- research probe and version-check tooling synchronized.

No change required:

- password/SMS auth hashing and bodies;
- places/access-controls/camera/public-camera/settings/finance paths;
- snapshot and SIP-credential response handling.

Follow-up diagnostic gap:

- the stock app distinguishes service-pipe/VPN/geoblock HTML from ordinary
  network failures; `http.py` currently reduces it to generic `ClientError`,
  and several API wrappers then return empty collections. See audit A-92.
