# PRD: Разговор по домофону (двусторонний звук) — будущая фича

> Статус: **спецификация на будущее** (вне скоупа FCM-event v1).
> Источник фактов: эксперимент `research/intercom-call-probe/` + [FINDINGS.md](FINDINGS.md)
> (live-проверка 2026-06-22, двусторонний звук **доказан рабочим**).
> Зафиксировано «пока свежо», чтобы не потерять hard-won знания.

## 1. Проблема и цель

После события вызова (FCM) логичный следующий шаг — **ответить на
вызов из HA и говорить с гостем у двери** (двусторонний звук), как это делает
приложение. Открытие двери уже есть (`accessControlOpen`), видео — через go2rtc.
Не хватает только аудио-разговора.

**Success criteria:**
- Пользователь из HA принимает входящий вызов домофона → слышит гостя и говорит ему.
- Открытие двери в разговоре (DTMF или REST) — есть.
- Работает **за NAT без проброса порта** (доказано: FreeSWITCH latching).
- Не ломает обычную работу: телефон продолжает звонить; HA «забирает» вызов
  только по явному действию пользователя (не авто).

## 2. Доказанные технические факты (фундамент, не гипотезы)

Всё проверено live (см. FINDINGS §«Медиа-слой»):
- **Бэкенд:** Kazoo (FreeSWITCH `mod_sofia` + Kamailio SBC).
- **SIP-креды:** `POST /rest/v1/places/{placeId}/accesscontrols/{acId}/sipdevices`
  `{installationId}` → `{login,password,realm}`, realm = `{acId}.intercom.{operator}.ru`.
- **Сигнализация:** REGISTER (Digest MD5) → форк INVITE на все зарег. устройства →
  **первый ответивший забирает** (остальным `CALL_END_ANSWERED_MOBILE`).
- 🔴 **INVITE — компактные заголовки** (`f/t/i/v/m`): валидный `200 OK` обязан
  копировать `From/To/Call-ID` дословно + эхо всех `Via`/`Record-Route`, иначе
  нет `ACK` → вызов не встаёт.
- **SDP:** audio-only `G.711 PCMU/PCMA` + `telephone-event`(DTMF) + `CN`, `ptime:20`.
  **Видео в SIP нет** (его покрывает go2rtc по `externalCameraId`).
- **Двусторонний RTP работает за symmetric NAT** благодаря RTP-latching
  FreeSWITCH (downlink приходит на наш публичный mapping). STUN — для публичного
  адреса в SDP `c=`; фиксированный локальный SIP-порт — против stale-регистраций.
- **Доказано:** answer → RTP оба направления (877 пакетов PCMU) → авто-открытие
  двери → `BYE`. Эталон кода — `probe_sip_media.py`.
- Модель — **2-сторонний bridge, first-answer-wins**; конференции/barge-in нет
  (несколько устройств одновременно в одном разговоре — недоступно у оператора).

## 3. Главный нерешённый вопрос (крутится вся фича)

**Где терминируется аудио — кто физически слышит/говорит?** HA headless, у него
нет «трубки». Это ключевой UX-выбор, его надо решить ДО архитектуры:
- **(A) Lovelace SIP-softphone card** (WebRTC в браузере/приложении HA) — юзер
  говорит через телефон/ПК, на котором открыт HA. Наиболее реалистично.
- **(B) HA Assist satellite / ESPHome voice** (умная колонка с микрофоном) —
  «ответить голосом» через колонку у двери квартиры.
- **(C) media_player + микрофон** — раздать downlink на колонку, uplink с микрофона.
- **(D) Только «прослушка» (one-way)** — слышать гостя в HA без ответа (проще,
  половина ценности).

## 4. Требования

**Функциональные:**
- F1: принять входящий вызов (SIP `200 OK` + медиа) по явному действию (кнопка/
  сервис `answer`), не авто.
- F2: двусторонний звук между гостем (дверь) и аудио-эндпоинтом HA (см. §3).
- F3: завершить вызов (`BYE`), сервис `hangup`.
- F4: открыть дверь в разговоре (DTMF `telephone-event` или REST `accessControlOpen`).
- F5: индикация активного вызова (binary_sensor/state «call active», окно по
  `CallInvalidated`).

**Нефункциональные:**
- N1: за NAT без проброса (latching + STUN), как приложение.
- N2: не «занимать» линию постоянно — регистрация транзиентная/по запросу,
  иначе панель показывает «Занято» (наблюдалось).
- N3: **настоящий SIP-стек** (PJSIP/`baresip`/`aiortc`), не ручной фреймер —
  корректные транзакции, NAT, медиа, кодеки.
- N4: секреты (SIP-пароль) не логировать.

## 5. Архитектурные опции (для будущего решения)

| Слой | Опции |
|---|---|
| SIP-стек | `aiortc` (Python, WebRTC/ICE — хорошо ложится на softphone-card), `baresip`/PJSIP (нативно, через sidecar/add-on), Asterisk add-on |
| Размещение | sidecar-сервис (как go2rtc), HA add-on, или in-HA (тяжело) |
| Аудио-эндпоинт | WebRTC → Lovelace card (A) — самый прямой путь с `aiortc` |

Предварительная гипотеза: **`aiortc`-based sidecar**, который держит SIP-регистрацию
(по запросу), принимает INVITE, и мостит медиа в **WebRTC** для Lovelace
softphone-card. go2rtc уже даёт WebRTC-видео — аудио-разговор рядом. Но это
гипотеза, решается на этапе design.

## 6. Фазирование

- **Фаза A** — событие вызова (FCM-event) — *в работе*.
- **Фаза B** — «прослушка» (one-way downlink в HA media_player) — дешевле, валидирует медиа-путь.
- **Фаза C** — полный двусторонний разговор (нужен аудио-вход, §3).
- Открытие двери — уже доступно (lock/`accessControlOpen`).

## 7. Риски
- Зависимость от RTP-latching оператора (если сменят SBC-конфиг — downlink сломается).
- Auto-answer/постоянная регистрация ломают обычную работу панели (N2).
- UX аудио-эндпоинта (§3) — главный риск принятия фичи.
