Статус: реализовано в `website/`. Этот документ — источник дизайн-решений сайта.
Сайт изолирован от интеграции: не трогает `custom_components/`, manifest, HACS и CI.

### Ограничения и честные оговорки (обязательны на сайте)

- API — неофициальное (зеркалирование мобильных приложений `myhome.proptech.ru`);
  оператор может его менять.
- Операторская ссылка live-видео живёт ~30 мин — долго открытый поток замирает
  by-design (A-71); go2rtc-публикация обновляет источник сама.
- Микрофон — только на HTTPS-origin; браузер не включает звук без жеста.
- Двусторонний звук — advanced-фича; помечать явно.
- Внешний RTSP: сетевая безопасность — ответственность пользователя.
- Камерная motion-история — отключённая по умолчанию сущность, включение
  запускает polling.
- Не называть интеграцию «официальной».

### Переиспользуемые активы

- **Реальный production-бандл карточки** `custom_components/elektronny_gorod/www/eg-intercom-call-card.js`
  (ESM, Lit вшит, содержит и `eg-event-history-card`). Сайт демонстрирует
  именно его — не перерисованную копию.
- **Паттерн demo harness** из `frontend/demo/index.html`: заглушки `ha-card` +

- Гостевое фото `pencil/images/generated-*.png` — сгенерированное, без PII.
- Скриншоты `docs/features/**/screenshots/*.png` (история, настенная панель).
- Официальные deep links my.home-assistant.io (HACS repository, config flow) —
  уже используются в README.

### Сегменты пользователей → задачи сайта

1. **HA-пользователь с HACS** → мгновенный CTA «Установить в HACS», deep link.
2. **Новичок с домофоном** → объяснение «что это» за 5 секунд, мастер установки
   от нуля («нет Home Assistant» — честный ответ, что нужно сначала).
3. **Продвинутый** → advanced-слой: события, сервисы, SIP, go2rtc, RTSP, сущности.
4. **Владелец настенной панели** → сцена wall-panel, blueprint-ы экрана вызова.

## 2. Концепция: «Путь одного звонка»

Одна страница = одна история: **реальный звонок в домофон проходит через весь
сайт**. Скролл — это движение сигнала: подъезд → FCM → Home Assistant →
устройства квартиры → замок → история → ваши автоматизации. Каждая секция —
станция на этом пути, и в каждой станции продукт показан «в действии», а не
описан.

Почему одна страница, а не многостраничник: все задачи пользователя (понять →
поверить → попробовать → установить) выстраиваются в одну траекторию; дробление
на страницы разорвало бы повествование и ослабило SEO малым объёмом каждой.
Глубокие ссылки — якоря (`/#demo`, `/#install`, `/#faq`), документация остаётся
в репозитории.

### Арт-дирекшн (по пунктам брифа)

1. **Главная визуальная идея** — «сигнальная нить»: непрерывная вертикальная
   линия-проводник, вдоль которой распространяется событие звонка. Она соединяет
   все секции; в узлах (домофон, HA, устройства) — импульсы.
2. **Почему подходит продукту** — продукт и есть событийная шина: FCM-событие
   реально проходит путь «дверь → облако → HA → устройства». Дизайн не
   иллюстрирует метафору, а честно показывает архитектуру.
3. **Сквозное прохождение** — нить стартует под hero (кнопка домофона), ведёт
   через сценарий звонка, playground, автоматизации (нить разветвляется на
   устройства), мастер установки (нить = шаги) и завершается в финальном CTA
   (замок открыт).
4. **Композиция** — асимметричная: контент-колонка смещена, нить живёт в левом
   поле на desktop и уходит за край на mobile; главный объект каждой сцены —
   реальный интерфейс карточки, крупно.
5. **Типографика** — «Unbounded» (variable, кириллица) только для H1/H2 и
   крупных цифр: широкий техно-дисплей, узнаваемость без логотипа. Текст и UI —
   «Golos Text»: нейтральный русский гротеск. Моно-акценты (entity_id, статусы) —
   ui-monospace.
6. **Интерфейс HA в дизайне** — production-бандл карточки вызова рендерится
   живьём в hero (автосценарий) и в playground (управляемый). Карточка не
   вставлена в мокап телефона по умолчанию: она — главный герой композиции;
   рамка устройства появляется только в переключателе «телефон/панель».
7. **Motion** — только причинно-следственный: импульс по нити → загорается
   следующий узел; фаза вызова → карточка меняет состояние; «дверь открыта» →
   зелёное подтверждение. `prefers-reduced-motion` отключает всё, кроме смены
   статических состояний. Ни одна анимация не блокирует interaction.
8. **Узнаваемость без логотипа** — сочетание: сигнальная нить + Unbounded +
   живая карточка вызова + фирменный приём «статусная строка событий» (лог
   `ring → answered → door_open` моноширинным под сценами).

### Цветовая система

- Light: почти белый `#f7f7f5`, ink-текст `#16181d`. Dark: графит `#101216`,
  текст `#e9eaec`. Тумблер + `prefers-color-scheme`.
- Акцент — технологичный синий `#0f8bd0` (родственник HA-синего `#03a9f4`
  из токенов карточки, углублён для контраста на светлом).
- Зелёный `#2f9e63` — только успех/принятие/дверь открыта. Красный `#d84a44` —
  только завершение и ошибки.
- Цвета операторов (зелёный ЭГ, красный Дом.ру) — локально: бейджи и
  переключатель совместимости.

## 3. Информационная архитектура

Одна страница, 10 станций + утилитарный футер:

1. `#hero` — живая сцена звонка (автосценарий на реальной карточке) + оффер + CTA.
2. `#scenario` — «Что происходит при звонке»: 6 шагов, скролл-повествование вдоль нити.
3. `#demo` — playground: фазы, темы, layout, устройство, RU/EN, действия.
4. `#features` — 12 жизненных ситуаций + раскрывающийся advanced-слой.
5. `#automations` — библиотека сценариев с YAML и копированием.
6. `#install` — мастер установки (ветвящиеся ответы → персональная инструкция).
7. `#compat` — проверка совместимости (единый конфиг-источник `src/data/compatibility.ts`).
8. `#trust` — открытый код, приватность, неофициальное API, поведение при сбоях.
9. `#faq` — реальные вопросы + путь «У меня не работает».
10. `#cta` — финальная трансформация + 4 действия.

## 4. Технический стек

**Vite + TypeScript + Lit** (без фреймворка страниц):

- Репозиторий уже живёт на Lit 3 + esbuild + vitest — та же ментальная модель,
  ноль новых парадигм для мейнтейнера.
- Карточка — custom element: статичный HTML + «острова» интерактивности
  подключают её напрямую; React/SSR добавили бы только трение.
- `index.html` — семантическая разметка всего контента (SEO без SSR),
  TS-модули прогрессивно оживляют hero, playground, мастер, checker.
- Сборка — статика в `website/dist`, публикуется на GitHub Pages / любой
  статический хостинг; `base: './'` — работает на поддомене и в подпапке.
- Данные сайта — типизированные модули `src/data/*` (единственный источник
  для совместимости, мастера, автоматизаций) + vitest-тесты на логику.
- Никаких секретов, серверов, обязательных cookies; аналитика — no-op
  абстракция `track()`.
